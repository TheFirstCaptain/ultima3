# B-001: Pascal String Search/Replace Uses Non-Portable Char Arithmetic

## Summary

`u3_search_replace` uses plain `char` for loop indexes and replacement-length arithmetic. Plain `char` signedness is implementation-defined in C, so shrinking replacements and long Pascal strings can behave differently across compilers or targets.

## Status

Resolved on 2026-06-02.

- `Core/src/u3_pascal_string.c` now uses explicit `int` arithmetic for `i`, `diff`, and `offset`.
- `harness/pascal_string_helpers_tests.c` covers shrinking replacement with a large tail and a match offset beyond the signed `char` range.
- Validation passed with the standard harness flags and with `-funsigned-char`.

## Observed Behavior

The portable core implementation currently declares `i`, `diff`, and `offset` as `char` in `Core/src/u3_pascal_string.c`.

If `char` is unsigned, a shrinking replacement such as replacing a 4-byte search string with a 1-byte replacement can make `diff` wrap to a large positive value. That value contributes to the `memmove` length used to shift the Pascal string tail.

If `char` is signed, long Pascal strings near the 255-byte Pascal string limit can overflow loop indexes such as `i`.

## Expected Behavior

The portable core should preserve the characterized legacy byte-level behavior while using explicit integer types for arithmetic so behavior does not depend on target `char` signedness or overflow.

## Reproduction

1. Inspect `Core/src/u3_pascal_string.c`.
2. Note that `u3_search_replace` uses plain `char` for `i`, `diff`, and `offset`.
3. Compile or reason under an unsigned-`char` target: negative `diff` values from shrinking replacements wrap before the `memmove` length is calculated.

## Investigation Notes

Independent review of the F-001 through F-006 portable core identified this as a portability defect.

Relevant implementation:

- `Core/src/u3_pascal_string.c`: `u3_search_replace`
- `harness/pascal_string_helpers_tests.c`: existing short-string coverage

The existing harness covers same-length, growing, shrinking, and terminal-match behavior on the local platform, but it does not cover unsigned-`char` behavior or long Pascal string lengths.

## Validation Plan

- Change `u3_search_replace` internal arithmetic to explicit signed/wide integer types such as `int16_t` or `int`.
- Preserve existing byte-level legacy behavior, including length-byte mutation and terminal-match quirks.
- Add harness coverage for shrinking replacement arithmetic that would fail under unsigned `char`.
- Add harness coverage for long Pascal strings with indexes greater than 127.
- Run:

```bash
make -C harness test
```

- If practical, add or manually run a targeted compile with unsigned `char` behavior and confirm the Pascal string helper tests still pass.
