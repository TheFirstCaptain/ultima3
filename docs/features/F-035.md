# F-035: First Overworld Location Transition Slice

## Objective

Detect and handle one narrow save-backed overworld location transition, such as entering a town, castle, dungeon entrance, or shrine tile, without implementing the full destination gameplay loop.

## User Value

After movement and terrain rules work, the next major sign of a playable world is that special map locations do something recognizable. A single transition slice proves the shell can move from raw overworld walking toward real game flow while keeping destination systems bounded.

## Non-Goals

- Do not implement every transition type in one feature.
- Do not implement full town, castle, dungeon, Ambrosia, combat, or NPC gameplay.
- Do not implement legacy menu/dialog flows unless the selected transition requires a minimal placeholder.
- Do not abandon the save-domain restriction that durable writes are characterized before broad autosave behavior.

## Dependencies

- F-031 provides save-backed party position and rendering.
- F-032 provides terrain/passability and explicit movement results.
- F-034 is recommended if the transition needs user-visible status or audio feedback.
- F-036 is recommended before this feature if the selected transition mutates durable state or changes save-safety rules.
- F-013 and F-015 provide resource and map fixture context for stable transition examples.

## Acceptance Notes

- One transition family and fixture position is selected and documented before implementation.
- The trigger rule is chosen before implementation: either movement onto the selected tile or an explicit command invokes the selected transition.
- The selected transition updates explicit portable state or returns a transition request.
- The shell handles the request with a bounded behavior, such as a status report, placeholder destination frame, or loading a characterized destination fixture.
- The current save document remains valid after the transition attempt.
- Harness coverage verifies both non-transition movement and selected transition detection.

## Affected Systems

- Legacy source files: `Sources/UltimaMain.c`, `Sources/UltimaMisc.c`, possibly `Sources/UltimaDngn.c`
- Portable core: likely `Core/include/u3_overworld.h`, `Core/src/u3_overworld.c`, and future location-state APIs if needed
- Modern shell: tick path, render refresh path, status path
- Modernization harness: `harness/overworld_movement_tests.c` or a new transition-focused harness test
- Documentation: feature tracker, selected transition contract, and manual validation notes

## Validation Plan

Add fixture-backed harness tests for selected transition detection, ordinary movement near the fixture, save-domain state validity, and shell translation of the transition request.

Run:

```bash
make -C harness test
swift test
rg -n "AppKit|SwiftUI|Metal|AVFoundation|Foundation|SDL|NS[A-Z][A-Za-z]+\\b|NSURL|Sandbox" Core
```

Manual validation:

1. Run `swift run Ultima3ModernShell`.
2. Load or create a current save with an active party.
3. Move to the documented transition fixture position.
4. Trigger the selected transition.
5. Confirm the shell reports or displays the bounded transition behavior.
6. Save and reload if the selected transition mutates durable state, and confirm the document remains valid.

## Open Questions

- Which transition should land first: town, castle, dungeon entrance, shrine, or another stable overworld tile?
- Should the first slice require an explicit command, or should movement onto the tile trigger it?
- Is a placeholder transition result acceptable, or should the first selected destination render a real fixture-backed frame?
