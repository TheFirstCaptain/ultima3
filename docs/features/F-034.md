# F-034: Overworld Command Feedback and Event Results

## Objective

Introduce a small portable event-result path for overworld commands so movement, blocked terrain, and first side effects can request consistent shell status text, redraws, and audio cues.

## User Value

Players need immediate feedback when a command succeeds, is blocked, or causes a rule effect. This feature makes the modern shell easier to validate manually and prepares it for richer gameplay without hard-coding rule interpretation in AppKit.

## Non-Goals

- Do not implement a full legacy text/message system.
- Do not implement NPC dialogue, shops, combat messages, or scripted map text.
- Do not make SwiftUI or AppKit own gameplay rule decisions.
- Do not replace the existing audio adapter; only route small portable event requests into it.

## Dependencies

- F-031 should provide save-backed movement.
- F-032 should provide blocked/passable movement results.
- F-033 may provide the first state-delta event. This feature can also land immediately after F-032 if status feedback becomes the blocking usability gap, but side-effect event coverage is then deferred until the first side-effect feature lands.

## Acceptance Notes

- Portable overworld command handling returns explicit event/result records for moved, blocked, redraw-needed, state-changed, and optional sound/status requests.
- The AppKit shell translates those records into existing game-host status text and audio adapter calls.
- Existing input and tick ownership remain unchanged: AppKit receives events, the input adapter queues portable commands, the tick path consumes commands and applies core results.
- Harness coverage verifies event records for success and blocked movement. If a side-effect feature has already landed, coverage also verifies at least one side-effect result.

Selected first result shape: `u3_overworld_move_result` carries portable `redraw`, `feedback`, `status`, and `sound_id` fields. Successful movement requests redraw, moved status, and `Step`; blocked movement requests blocked status and `Bump`. State-delta fields wait for F-033.

## Affected Systems

- Portable core: `Core/include/u3_overworld.h`, `Core/src/u3_overworld.c`, possibly `Core/include/u3_audio.h`
- Modern shell: `ModernShell/Sources/AppDelegate.swift`, `ModernShell/Sources/ShellInputAdapter.swift`, `ModernShell/Sources/ShellAudioAdapter.swift`
- Modernization harness: `harness/overworld_movement_tests.c`, `harness/input_adapter_tests.c`, possibly `harness/audio_adapter_tests.c`
- Documentation: feature tracker and `ModernShell/README.md`

## Validation Plan

Add harness tests for emitted result records and Swift tests for shell translation where the adapter logic is factored. Keep the core API portable and scan boundaries after implementation.

Run:

```bash
make -C harness test
swift test
rg -n "AppKit|SwiftUI|Metal|AVFoundation|Foundation|SDL|NS[A-Z][A-Za-z]+\\b|NSURL|Sandbox" Core
```

Manual validation:

1. Run `swift run Ultima3ModernShell`.
2. Load or create a current save with an active party.
3. Move onto passable terrain and confirm the status reports a successful command/result.
4. Attempt movement into blocked terrain and confirm the status reports a distinct blocked result.
5. Confirm any selected sound cue is routed through the existing shell audio adapter.

## Open Questions

- Resolved for this slice: status text remains compact debug wording (`Move ...`, `Blocked ... tile ...`) instead of mapping to full legacy text.
- Resolved for this slice: the result shape is limited to overworld movement feedback fields and can be generalized later if town, dungeon, or combat command paths need a shared dispatcher.
- Resolved for this slice: audio requests live in overworld result records as portable `sound_id` values and the shell translates them into the existing audio adapter queue.
