# F-055: Combat Flee and Defeat Flow

## Objective

Add bounded combat escape and party-defeat outcomes, including explicit shell restoration or failure handling.

## User Value

Combat needs more than victory as an exit condition. Players should be able to attempt escape, and the shell should handle an incapacitated party without leaving combat in an undefined state.

## Non-Goals

- Do not implement every legacy morale, pursuit, or special-case escape branch if a narrower first escape policy is needed.
- Do not implement death/ash recovery, resurrection, or healer services.
- Do not implement autosave or in-combat save support.
- Do not add a polished game-over presentation beyond bounded status/render restoration.

## Dependencies

- F-052 provides combat victory/source-session restoration patterns.
- F-051 provides monster mutations that can incapacitate party members.
- F-036 defines manual save policy and unsafe-state rejection.

## Acceptance Notes

- A Flee command returns explicit portable success/failure results using deterministic test inputs and runtime rolls in the shell.
- Successful flee discards transient combat state and restores the preserved source session under the existing save policy.
- Failed flee remains in combat, reports status/audio, and continues through the bounded monster response if appropriate.
- All-active-party defeated or invalid living-party state reports an explicit defeat outcome without unsafe actions.
- Party HP/status mutations remain in the current in-memory document and become durable only after reaching a save-eligible Sosaria state.

## Implementation Notes

- Implemented first escape surface is the explicit combat command `F`.
- The portable core reports flee success when the supplied roll is at least `128`; runtime shell rolls are random and tests inject deterministic rolls.
- Successful flee plays Step feedback, discards the transient combat session, and restores the preserved dungeon source session without XP or combat loot.
- Failed flee plays Bump feedback, stays in combat, advances the active character, and runs the bounded one-monster response.
- `u3_combat_check_party_defeat` reports all active party members defeated using the same active-party size boundary as combat setup.
- After a monster turn defeats the last active party member, the shell writes HP/status mutations to the current in-memory document, restores the preserved dungeon source session, and keeps Save rejected until Sosaria.

## Affected Systems

- Legacy source files: `Sources/UltimaSpellCombat.c` flee and party death/cleanup paths
- Portable core: flee/defeat result contract
- Modern shell: combat exit routing and source-session restoration
- Save adapter: party status/HP mutation handling
- Modernization harness and Swift tests: flee success/failure, defeat detection, restoration, save policy
- Documentation: `ModernShell/README.md`, feature tracker

## Validation Plan

Cover flee success, flee failure, active combat retention after failed flee, source dungeon restoration after successful flee, all-party-defeated detection, dirty-state behavior, and save rejection until Sosaria. Run `make -C harness test`, `swift test`, and the Core boundary scan.

Manual validation: enter combat, attempt flee with success and failure cases, confirm status/render restoration or combat retention, then confirm Save remains rejected until returning to Sosaria.
