# F-024: Party/Roster State Adapter

## Objective

Expose modern save-domain party and roster state through a portable adapter suitable for shell inspection and future setup flows.

## User Value

Players need party and roster data to become visible in the modern app before character creation, party selection, and save workflows can be built cleanly.

## Non-Goals

- Do not implement the full character creation UI.
- Do not mutate legacy roster resources in place.
- Do not put SwiftUI or AppKit types in `Core/` state contracts.
- Do not broaden the persistence format beyond F-014/F-020 decisions.

## Dependencies

- F-020C should define the native save fixture.
- F-020E may inform legacy roster import behavior.
- F-021 should provide a minimal loadable save-domain state.

## Acceptance Notes

- Party and roster state can be read through a portable interface.
- Shell or inspector UI can display the state without owning persistence serialization.
- Harness coverage validates basic empty/default/loaded roster shapes.
- The adapter leaves mutation rules explicit for later character creation work.

## Open Questions

- Should first shell visibility be a debug inspector, a SwiftUI preferences-style panel, or a simple menu-triggered status readout?
- Which roster fields are required for the first gameplay smoke versus later full creation flows?

## Status

Proposed candidate after F-021.
