# F-024: Party/Roster State Adapter

## Objective

Expose modern save-domain party and roster state through a portable adapter suitable for shell inspection and future setup flows.

## User Value

Players need party and roster data to become visible in the modern app before character creation, party selection, and save workflows can be built cleanly.

## Non-Goals

- Do not implement the full character creation UI.
- Do not mutate legacy roster resources in place.
- Do not put SwiftUI or AppKit types in `Core/` state contracts.
- Do not broaden the persistence format beyond F-014/F-020 decisions.

## Dependencies

- F-020C should define the native save fixture.
- F-020E may inform legacy roster import behavior.
- F-021 should provide a minimal loadable save-domain state.

## Acceptance Notes

- Party and roster state can be read through a portable interface.
- Shell or inspector UI can display the state without owning persistence serialization.
- Harness coverage validates basic empty/default/loaded roster shapes.
- The adapter leaves mutation rules explicit for later character creation work.

## Implementation Notes

- Added `u3_party_load_summary` as a portable core adapter over the F-021 save-domain state.
- The decoded first-slice summary includes active party size, active roster ids, roster occupancy, name, status, race, class, sex, HP/max HP, level, food, and gold.
- The shell exposes the state through `Game > Inspect Party/Roster` and reports it in the existing status overlay.
- Harness coverage validates default resource-backed party data, empty roster shape, and malformed active party ids.
- Swift shell tests validate the user-facing default party readout and missing-resource fallback.
- Independent review findings were resolved by requiring active roster ids to reference occupied roster entries, adding food/gold assertions, and covering non-slot-1 leader formatting.

## Open Questions

- Resolved for first implementation: expose shell visibility as a simple `Game > Inspect Party/Roster` menu-triggered status readout, not a SwiftUI panel.
- Resolved for first implementation: decode only stable summary fields: active party size and roster ids, roster occupancy, name, status, race, class, sex, HP/max HP, level, food, and gold. Leave inventory, equipment, spells, and mutation rules for later setup/character-creation slices.
