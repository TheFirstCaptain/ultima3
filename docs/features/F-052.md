# F-052: Combat Victory and Return Flow

## Objective

Detect first combat victory, apply bounded rewards/state cleanup, and return from combat to the preserved source session.

## User Value

The first combat loop needs a clear finish. Players should be able to complete a simple encounter and resume dungeon or world exploration with intended party/roster mutations preserved.

## Non-Goals

- Do not implement every loot table, chest/reward branch, spell aftermath, or flee behavior.
- Do not support every combat source if dungeon-origin combat is enough for the first victory slice.
- Do not add autosave or in-combat save support.
- Do not implement polished combat victory presentation beyond bounded status/audio/render restoration.

## Dependencies

- F-048 provides combat source-session preservation.
- F-049 through F-051 provide renderable combat state and turn mutation.
- F-009A extracted monster damage/death reward behavior.
- F-036 defines manual save policy.

## Acceptance Notes

- Combat session detects when all active monsters are defeated and returns an explicit victory result.
- Bounded reward mutations selected for this slice apply to the current in-memory native document and mark it unsaved.
- The shell restores the preserved source session after victory: dungeon session if combat began in dungeon, with Sosaria context still retained underneath.
- Save remains rejected if the restored source is still dungeon/town mode, and becomes available only after returning to Sosaria under the existing policy.
- Combat session data is discarded after victory and is not written into the native save document.
- Failed restoration reports explicit status without losing the current document.

## Affected Systems

- Legacy source files: `Sources/UltimaSpellCombat.c` `Victory`, `DamageMonster`, combat cleanup and return paths
- Portable core: victory/result contract and reward helpers
- Modern shell: combat-to-source session restoration
- Save adapter: reward mutation application and dirty-state behavior
- Modernization harness and Swift tests: victory detection, rewards, return-session integrity, save policy
- Documentation: `ModernShell/README.md`, feature tracker

## Validation Plan

Cover victory detection, reward mutation bytes, dirty-state marking, source dungeon restoration, combat-session discard, save rejection until Sosaria, failed restoration handling, and save/reload after returning to Sosaria. Run `make -C harness test`, `swift test`, and the Core boundary scan.

Manual validation: enter a controlled combat, defeat the final monster, confirm return to the prior dungeon/source frame, return to Sosaria, save, reload, and confirm intended rewards/mutations persisted.

- Should combat victory audio/music be added here or deferred to a later presentation pass?
