# F-001D: Characterization Candidate Analysis

## Objective

Recommend the first legacy behaviors suitable for characterization tests or small harness executables.

## User Value

This turns inventory into an actionable modernization starting point without immediately porting the full app.

## Non-Goals

- Do not implement tests yet.
- Do not port candidate logic.
- Do not require the legacy app to run.

## Acceptance Notes

- At least three candidate behaviors are identified.
- Each candidate includes rationale, likely files, inputs/outputs, platform coupling, and risk.
- One recommended first target is named.
- `docs/modernization/validation-strategy.md` is updated with candidates.

## Affected Systems

- Legacy source files: to be identified by prior analyses
- Modernization harness: validation strategy
- Resources/assets: possible fixtures only, no changes
- Build or tooling: none expected
- Documentation: `docs/modernization/validation-strategy.md`, optionally `docs/modernization/porting-strategy.md`

## Validation Plan

Review outputs from F-001A, F-001B, and F-001C. Confirm candidates have clear observable behavior and manageable platform coupling.

## Open Questions

- Is text/resource behavior, combat calculation, dungeon movement, or map logic the best first target?
- What fixture data is needed for the first characterization harness?
