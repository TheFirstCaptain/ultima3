# F-050: First Player Combat Command Slice

## Objective

Handle the first bounded set of player combat commands in an active combat session: movement, pass, and one direction-supplied attack path.

## User Value

Combat becomes interactive. Players can issue basic tactical commands and see immediate state/status feedback without needing the full spell, item, or victory systems.

## Non-Goals

- Do not implement spells, ranged weapon inventory complexity beyond the selected first attack path, autocombat, flee, or victory.
- Do not implement monster turns in this slice except as explicitly deferred status.
- Do not add modal blocking input loops.
- Do not make combat state durable.

## Dependencies

- F-048 provides combat session entry.
- F-049 provides combat arena rendering and occupant state.
- F-009B extracted player combat attack resolution after direction selection.
- F-018 and F-023 provide queued input and fixed tick routing.

## Acceptance Notes

- The shell routes combat-mode keyboard commands through a combat-specific command handler without taking ownership away from AppKit.
- Basic movement/pass updates transient combat session state and redraws through portable results.
- Attack command requests and consumes a cardinal direction in a bounded non-blocking flow.
- Direction-supplied attack uses or adapts the existing portable `CombatAttack` contract and reports hit/miss/deferred outcomes.
- Any roster or inventory mutations selected for this slice apply only to the current in-memory document, mark it unsaved, and remain subject to combat/dungeon save rejection until return to Sosaria.
- Unsupported commands report explicit deferred status without corrupting the session.

## Affected Systems

- Legacy source files: `Sources/UltimaSpellCombat.c` `Combat`, `CombatAttack`, command handling, movement helpers
- Portable core: combat command/result contract and any missing mutation helpers
- Modern shell: combat input routing and prompt state
- Audio/render adapters: Step/Bump/Attack/Hit feedback and redraw mapping
- Modernization harness and Swift tests: command routing, direction prompt, hit/miss, mutation/no-mutation cases
- Documentation: `ModernShell/README.md`

## Validation Plan

Cover movement, blocked movement, pass, attack direction prompt, valid/invalid direction, hit/miss deterministic rolls, redraw/status/audio fields, and unsupported commands. Run `make -C harness test`, `swift test`, and the Core boundary scan.

Manual validation: enter a combat session, move/pass, start an attack, provide a direction, confirm status/audio/redraw behavior, and confirm Save remains rejected while combat is active.

## Open Questions

- Which exact legacy command keys should be included in the first player-command subset?
- Should the first attack path require a melee weapon only, or should dagger/projectile behavior be included because F-009B already characterized it?
