# F-038: Two-Dimensional Location Navigation

## Objective

Add portable movement and centered rendering for loaded town and castle sessions, using destination map size and passability rules instead of Sosaria wrapping and save-backed overworld coordinates.

## User Value

Players should be able to walk around an entered location rather than seeing a static fixture. This creates a reusable 2D gameplay path for towns and castles before adding their interactions.

## Non-Goals

- Do not return to Sosaria in this feature.
- Do not implement NPC movement, talk, shops, stealing, combat, or castle-specific rules.
- Do not mutate the persistent Sosaria map or position.
- Do not reuse overworld wrapping when destination boundaries are intentionally exits.

## Dependencies

- F-037 provides validated 2D location sessions and destination frames.
- F-031 through F-034 provide the existing input, result, feedback, turn, and viewport patterns.

## Acceptance Notes

- Cardinal movement operates on the loaded map dimensions and explicit location position.
- Destination movement has characterized tile passability independent of the narrower non-ship Sosaria rule.
- Map edges do not wrap. A move that reaches an exit boundary returns an explicit exit-request result for F-039 rather than silently changing location mode.
- Movement results request redraw, status, and Step/Bump audio through portable fields.
- The centered viewport follows the location-session position and uses the existing render contract.
- Location movement does not write `PRTY` Sosaria position bytes.
- Handled location commands apply the shared `PRTY` move-counter cost characterized in F-033, because the legacy game loop calls `Routine6E35` and `IncMoves` in towns and castles as well as on Sosaria. The result exposes the delta and marks the in-memory document dirty for later outdoor save.

## Affected Systems

- Legacy source files: `Sources/UltimaMain.c`, especially movement, `ValidDir`, and `Routine6E35`
- Portable core: new 2D location navigation module or a clearly separated mode in the overworld module
- Modern shell: tick dispatch and location-frame refresh
- Modernization harness: focused location navigation tests with bundled and synthetic fixtures
- Documentation: `ModernShell/README.md`

## Validation Plan

Cover passable and blocked destination tiles, all cardinal directions, non-wrapping boundaries, exit requests, destination viewport centering, shared move-counter cost, and no mutation of Sosaria position/map bytes. Add Swift tests for tick-driven frame and feedback updates.

Run the standard harness, Swift tests, and Core boundary scan.

Manual validation: load the selected town fixture, walk in all four directions, confirm walls block, confirm the viewport follows the party, and confirm reaching an edge reports a pending exit without corrupting the current save.

## Open Questions

- Resolved for this slice: loaded `MAPS` bytes already contain the stationary NPC fixture tiles used by legacy `GetXYVal`. Movement evaluates those bytes directly, so NPC/object tiles at or above the blocking threshold remain blocked without a second `MONS` collision rule. Autonomous NPC movement remains deferred.
- Resolved for this slice: towns and castles use the same legacy non-ship `ValidDir` movement classification. Forcefield and lava remain passable for movement, but their damage and effect events are deferred.
- Resolved for this slice: reaching `x == 0` or `y == 0` requests the legacy location exit. Other out-of-range movement is blocked and never wraps. F-039 will consume the exit request and restore Sosaria.
