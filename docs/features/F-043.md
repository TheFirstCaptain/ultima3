# F-043: Dungeon Navigation and Surface Return

## Objective

Wire the extracted dungeon movement, turning, climb, and descend rules into the shell tick path, redraw the perspective frame after commands, and restore the pre-entry Sosaria session when the dungeon exits.

## User Value

This completes the first playable dungeon loop: enter, orient, navigate, change levels where allowed, and return safely to the overworld.

## Non-Goals

- Do not implement dungeon combat, traps, chests, fountains, marks, gremlins, or NPC dialogue.
- Do not implement torch inventory/decay beyond any minimum visibility policy explicitly selected for this slice.
- Do not persist while in a dungeon or add autosave timing.
- Do not bypass AppKit input ownership with a modal loop.

## Dependencies

- F-041 provides dungeon session entry and retained Sosaria context.
- F-042 provides perspective redraws.
- F-008 provides portable dungeon navigation transitions.
- F-023 provides the deterministic shell tick.

## Acceptance Notes

- Arrow/numeric commands map to forward, retreat, left turn, and right turn using dungeon semantics rather than overworld cardinal semantics.
- Descend and Klimb commands use the existing portable ladder validation and level transitions.
- Every handled command returns explicit moved/turned/blocked/level-changed/exit/redraw/feedback fields.
- A valid dungeon-exit result restores party location mode zero, the exact retained Sosaria position and frame, and overworld command routing.
- Blocked or invalid dungeon commands never corrupt the active session or durable save document.
- Save remains rejected during the dungeon session and works after successful surface restoration.

## Affected Systems

- Legacy source files: `Sources/UltimaDngn.c` navigation loop; `Sources/UltimaMisc.c` `Routine6E6B`
- Portable core: existing `u3_dungeon` result contracts and any bounded event extensions
- Modern shell: mode-aware input dispatch, tick processing, feedback, and session restoration
- Modernization harness and Swift tests: command routing and full round-trip coverage
- Documentation: `ModernShell/README.md`

## Validation Plan

Cover all navigation commands, turns, blocking, valid/invalid level changes, top-level exit, feedback/redraw fields, save restriction, exact Sosaria restoration, and repeated entry/exit cycles. Run the standard harness, Swift tests, and boundary scan.

Manual validation: enter the documented dungeon, turn and move through a known corridor, test a blocked move, use a valid ladder if reachable, exit to the surface, confirm the original Sosaria location, then save and reload.

## Open Questions

- For an unlit dungeon, should the first playable slice grant a session-scoped diagnostic light, require a later Ignite feature, or render only status until torch behavior lands?
- Should a top-level Klimb command exit immediately as in the extracted core, or require an explicit confirmation in the shell?
