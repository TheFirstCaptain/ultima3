# F-032: First Overworld Gameplay Interaction Slice

## Objective

Implement the first narrow overworld gameplay interaction after save-backed movement, starting with terrain/passability and movement side effects before map transitions.

## User Value

Once movement is save-backed, gameplay needs rules that make Sosaria feel meaningful. A small terrain interaction slice gives the shell visible rule enforcement without pulling in the full town, dungeon, combat, and encounter systems at once.

## Non-Goals

- Do not implement all overworld commands.
- Do not implement full town, dungeon, castle, combat, or Ambrosia transitions.
- Do not implement complete monster movement or encounter generation unless explicitly chosen as the first interaction.
- Do not hand-roll platform behavior in the shell; rules belong behind portable core APIs.

## Dependencies

- F-031 should provide save-backed overworld position and rendering.
- F-030 should provide current save mutation and writeback.
- F-013 and F-015 provide resource and map fixture context.

## Acceptance Notes

- One interaction family is selected and documented before implementation, with terrain/passability as the preferred first target.
- The portable core returns explicit movement or interaction results for shell rendering, status, and audio adapters.
- Harness coverage characterizes the selected legacy behavior against known map tiles or fixtures.
- Manual shell validation confirms a visible moved/blocked/effect distinction.

Selected slice: non-ship overworld terrain/passability from the legacy `ValidDir` branch. Water (`0`), tile `16`, and tiles `>= 48` block ordinary foot movement; low terrain below `48` except `0` and `16` is passable. Forcefield (`128`), lava (`132`), moongate (`136`), and mark (`248`) are treated as passable for movement only; their damage, sound, and special effects remain deferred.

## Affected Systems

- Legacy source files: `Sources/UltimaMain.c`, `Sources/UltimaMisc.c`, `Sources/UltimaMacIF.c`
- Portable core: likely `Core/include/u3_overworld.h`, `Core/src/u3_overworld.c`, plus any new domain module if the interaction grows beyond movement
- Modern shell: input tick path, render refresh path, audio/status adapter path
- Modernization harness: `harness/overworld_movement_tests.c` or a new focused harness test
- Documentation: feature tracker, validation notes, and any decision log entry if a durable behavior choice is made

## Validation Plan

Add fixture-backed harness tests for the selected interaction family. For terrain/passability, cover passable movement, blocked movement, boundary behavior, and any explicit side-effect result such as sound or status text request.

Run:

```bash
make -C harness test
swift test
rg -n "AppKit|SwiftUI|Metal|AVFoundation|Foundation|SDL|NS[A-Z][A-Za-z]+\\b|NSURL|Sandbox" Core
```

Manual validation:

1. Run `swift run Ultima3ModernShell`.
2. Load or create a current save with an active party.
3. Move onto known passable terrain and confirm movement succeeds.
4. Attempt movement into known blocked terrain and confirm position does not change.
5. Confirm the shell reports the distinction through status and redraw.
