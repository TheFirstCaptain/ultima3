# F-008: Dungeon Navigation Core

## Objective

Characterize and extract bounded dungeon navigation behavior into the portable core.

## User Value

Dungeon movement is a focused gameplay slice with visible state transitions: turning, moving forward or backward, changing dungeon levels, and exiting. Extracting it after the autocombat work broadens the portable core beyond combat helpers while preserving behavior from the legacy implementation.

## Non-Goals

- Do not port dungeon rendering, dungeon graphics, sound, text output, or input polling.
- Do not rewrite the full dungeon loop.
- Do not convert dungeon data files or resource loading.
- Do not modify legacy `Sources/` unless an adapter step is explicitly chosen later.
- Do not change behavior captured from `Sources/UltimaDngn.c`.

## Acceptance Notes

- Legacy dungeon navigation functions are inspected and characterized, primarily `Forward`, `Retreat`, `Left`, `Right`, `dDescend`, and `dKlimb`.
- Portable API uses explicit caller-owned dungeon position, heading, level, and exit state.
- Harness tests cover turning, forward movement, retreat movement, blocked movement, level descent, level climb, and dungeon exit behavior where applicable.
- Rendering, messages, and sound are represented only by testable return values or omitted side effects.
- `make -C harness test` passes.

## Result

Status: complete as of 2026-05-31.

Added:

- `Core/include/u3_dungeon.h`
- `Core/src/u3_dungeon.c`
- `harness/dungeon_navigation_tests.c`

Extended:

- `harness/Makefile`

The portable dungeon navigation core uses explicit caller-owned `u3_dungeon_state` for dungeon bytes, position, heading, level, and exit state. Navigation functions return `u3_dungeon_result` with state-transition and side-effect flags instead of calling rendering, sound, or text helpers.

Characterized behavior includes one-tile coordinate wrapping for reads and movement, unwrapped dungeon writes, forward and retreat movement, exact `0x80` movement blocking, turn blocking on current tiles `>= 0xA0`, down-ladder validation with bit `0x20` on non-wall tiles, bottom-level descend rejection to preserve core bounds, up-ladder validation with bit `0x10`, redraw signaling, invalid-command signaling, no-go signaling, and level-zero climb exit behavior.
