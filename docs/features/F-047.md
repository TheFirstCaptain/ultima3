# F-047: Dungeon Interactive Tile Flows

## Objective

Add bounded shell-supported interaction flows for dungeon tiles that require player choice or repeated prompts: fountains, marks, Time Lord messages, and dungeon chests.

## User Value

Interactive dungeon fixtures are the next step from navigation and passive hazards toward a playable dungeon loop. They exercise party-member selection, inventory/HP/status mutation, and longer message presentation without committing to the full combat loop.

## Non-Goals

- Do not implement shops, town dialogue branching, spell casting, or full combat.
- Do not implement every chest reward/loot policy if a narrower first chest result is needed for validation.
- Do not add in-dungeon save support or autosave.
- Do not migrate SwiftUI into primary game command ownership.

## Dependencies

- F-043 provides dungeon session routing and return.
- F-044 provides light and command-state handling in dungeon mode.
- F-046 establishes special tile effect contracts and shared result/event mapping.
- F-024 and F-029 expose roster and active party data for character selection and mutation.

## Acceptance Notes

- The shell has a bounded, mode-aware way to request a party-member choice for dungeon interactions without blocking the AppKit tick loop.
- Fountain effects preserve the characterized legacy variants keyed by `xpos & 0x03`: poison, heal, damage, and cure poison.
- Mark/brand effects apply the correct bit to the selected roster entry, apply HP cost, and report feedback.
- Time Lord and similar message-only interactions display bounded text and restore dungeon rendering cleanly afterward.
- Dungeon chest interaction recognizes current-cell chest state, applies a first characterized open/remove/reward policy, and reports deferred cases explicitly.
- Party and roster mutations from fountains, marks, chest rewards, inventory costs, HP, and status changes apply to the current in-memory native document, mark it as unsaved, and become durable only after the party returns to Sosaria and the existing manual Save command succeeds.
- Dungeon-session tile mutations such as consumed chest or interaction cells apply only to the transient dungeon map/session copy and are not written to the native save document in this slice.
- Canceled or invalid selections leave state unchanged except for any legacy-compatible prompt feedback selected for the slice.

## Affected Systems

- Legacy source files: `Sources/UltimaDngn.c` `dngnotcombat` cases 1, 2, and 5; `Sources/UltimaMain.c` `GetChest`
- Portable core: dungeon interactive-effect contracts and party/roster mutation helpers
- Modern shell: non-modal prompt state, status/message presentation, redraw/audio mapping
- Modernization harness and Swift tests: selected-character flows, mutation bytes, cancel paths, chest state
- Documentation: `ModernShell/README.md`

## Validation Plan

Cover each fountain variant, dead/invalid/canceled character selection, mark bit setting and HP cost, message-only interactions, first chest open behavior, current-document dirty-state marking for party/roster mutations, transient-only dungeon tile mutation, redraw/status/audio fields, and save policy. Run `make -C harness test`, `swift test`, and the Core boundary scan.

Manual validation: enter the dungeon, trigger each supported interactive tile from a controlled fixture or reachable location, cancel and complete character selections, inspect pending party/roster changes through the shell, return to Sosaria, save, reload, and confirm only party/roster changes intended for the current native document were retained.

## Open Questions

- Should the first interactive prompt be keyboard-only for parity with legacy flow, or a small SwiftUI non-game panel consistent with existing setup boundaries?
- Which chest behavior is narrow enough for this slice while still matching the legacy dungeon path users will hit first?
