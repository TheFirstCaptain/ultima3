# F-063: Overworld Encounter and Transport Expansion

## Objective

Add the next Sosaria systems around random encounters, terrain hazards, and transport-aware movement such as ships and horses.

## User Value

Sosaria movement is currently safe and bounded, but real play depends on danger, transport, and terrain effects. This tranche makes outdoor travel behave more like the game while still preserving testable rules.

## Non-Goals

- Do not implement every vehicle, cannon, whirlpool, moongate, mark, or quest interaction in one pass.
- Do not add full sea combat or special event chains unless selected as a bounded slice.
- Do not change town/dungeon session persistence policy.
- Do not couple random encounter decisions to AppKit or runtime-only state.

## Dependencies

- F-031 through F-034 provide save-backed overworld movement, terrain passability, turn accounting, and feedback.
- F-048 provides combat session entry mechanics from dungeon encounters that can inform overworld encounter entry, but this feature must define Sosaria-origin combat restoration explicitly instead of inheriting dungeon-origin exit policy.
- F-049 through F-055 provide enough transient combat loop behavior for encounter validation.

## Acceptance Notes

- A selected outdoor encounter path can request and initialize a transient combat session from deterministic rolls and validated resources.
- Outdoor-origin combat snapshots the exact Sosaria party position, map frame, and save-eligible mode before combat entry. Victory and successful flee discard transient combat state and restore that Sosaria context; defeat applies the selected HP/status mutations, restores Sosaria context, and remains save-eligible under the outdoor save policy unless a later decision documents a stricter rule.
- Outdoor combat exit results explicitly report whether rewards, XP, HP/status changes, dirty-state marking, and save eligibility differ from dungeon-origin combat exits.
- At least one terrain or transport rule is added with explicit passability, mutation, audio/status, and redraw behavior.
- Randomness is injectable for harness and Swift smoke tests.
- Movement and encounter outcomes update the current in-memory native document only for intended party/roster mutations and remain save-safe under the existing Sosaria policy.
- Unsupported transport or terrain cases report explicit deferred statuses without corrupting party position.

## Affected Systems

- Legacy source files: overworld movement, encounter, ship/horse, and terrain-effect paths
- Portable core: overworld movement result contract, encounter request, transport state helpers
- Modern shell/resource adapter: encounter session loading, deterministic roll injection, status/audio mapping
- Modernization harness and Swift tests: deterministic encounter, terrain, transport, and no-corruption coverage
- Documentation: `ModernShell/README.md`, feature tracker

## Validation Plan

Cover deterministic no-encounter and encounter rolls, combat-session initialization from Sosaria, victory/flee/defeat restoration to the exact Sosaria context, save eligibility after outdoor-origin combat exits, party-position preservation on failed encounter setup, selected terrain/transport movement rules, document mutation and no-mutation cases, and save/reload after outdoor movement. Run `make -C harness test`, `swift test`, and the Core boundary scan.

Manual validation: travel outdoors, trigger a deterministic or debug-assisted encounter, resolve victory/flee/defeat paths as available, confirm return to the exact Sosaria position and save eligibility, exercise the selected transport/terrain rule, save/reload, and inspect party position/state.

## Open Questions

- Should outdoor combat reuse Dungeon Doom's first combat screen fixture initially or characterize dedicated overworld combat screens first?
- Which transport slice carries the best value: horse movement, ship embark/disembark, or terrain hazard effects?
- Do we need a debug/test menu command to force deterministic outdoor encounters for manual validation?
