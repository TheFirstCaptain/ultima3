# F-023: Shell Game Loop Tick

## Objective

Introduce a minimal modern shell tick that coordinates input polling, state update, render refresh, and queued audio dispatch.

## User Value

The modern app needs a predictable runtime heartbeat before small gameplay slices can feel interactive instead of menu-driven.

## Non-Goals

- Do not port the full legacy main loop.
- Do not make SwiftUI own the primary game loop.
- Do not add broad gameplay simulation beyond a deterministic smoke update.
- Do not make `Core/` depend on AppKit timers, run-loop APIs, or platform clocks.

## Dependencies

- F-017 provides the renderer adapter path.
- F-018 provides the input queue.
- F-019 provides the audio event queue.
- F-020 and F-021 should clarify resource/save state entry points.

## Acceptance Notes

- The AppKit shell owns the concrete timer or display-link mechanism.
- Core-facing update inputs are portable values or explicit events.
- A smoke tick can consume input, update minimal state, request a render frame, and dispatch queued audio.
- Manual HIL verification confirms that the shell remains responsive while the tick is active.

## Open Questions

- Resolved for first implementation: use a fixed AppKit-owned timer that starts on launch, with a modest smoke cadence around 4 Hz. A debug menu item may pause or resume the timer, but the first heartbeat should not require manual tick commands.
- Resolved for first implementation: mutate only a tiny portable smoke tick state, such as tick count, phase, and consumed event counters. Do not add movement, party mutation, map mutation, combat orchestration, or legacy timing semantics in this slice.

## Status

Approved for implementation on 2026-06-07.

Completed on 2026-06-07.

Implemented a portable `u3_tick` smoke state with tick count, phase, consumed-input count, dispatched-audio count, and explicit tick results. Added harness coverage for initialization, phase advancement, render-refresh requests, event counters, and invalid inputs.

The modern shell now starts an AppKit-owned 0.25-second timer on launch. Keyboard, mouse, menu, and test-sound actions enqueue events; `ShellSmokeState.runTick()` consumes at most one input event and one audio event per tick, advances the portable tick state, updates tick status, and lets the existing published status repaint the game host. `Game > Toggle Tick` pauses and resumes the timer.

Automated validation completed:

```bash
swift package clean
swift build --product Ultima3ModernShell
swift test
make -C harness test
rg -n "AppKit|SwiftUI|Metal|AVFoundation|Foundation|SDL|NS[A-Z][A-Za-z]+\\b|NSURL|Sandbox" Core
```

Independent subagent review completed with no findings.

Manual HIL validation completed by the user:

1. Tick status increments at about 4 Hz and the shell remains responsive.
2. Keyboard/click input is queued and consumed on the next tick.
3. `Game > Test Sound` dispatches audio on tick; the overlay reports `Audio Sound Error1` when the test sound plays successfully.
4. `Game > Toggle Tick` pauses and resumes status advancement.
