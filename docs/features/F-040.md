# F-040: First Town NPC Talk Slice

## Objective

Add one bounded Talk interaction in the loaded town session by resolving an adjacent NPC from `MONS` data and returning its characterized text from the matching `TLKS` record.

## User Value

Walking through a town becomes meaningful when its inhabitants can respond. This proves the resource-backed interaction path needed for later shops, hints, and quests.

## Non-Goals

- Do not implement every legacy talk opcode, branching conversation, shop, pub, bribe, or quest mutation.
- Do not implement autonomous NPC movement.
- Do not add a general rich-text or dialogue-window framework.
- Do not let AppKit interpret raw gameplay tables.

## Dependencies

- F-037 provides compatible `MONS` and `TLKS` data in the location session.
- F-038 provides location position and occupancy-aware navigation context.
- F-039 provides a real town session in the shell.

## Acceptance Notes

- The Talk command is routed through the existing input/tick ownership path.
- A portable interaction request identifies an adjacent NPC using explicit direction input or a documented first-slice targeting rule.
- Core/resource helpers resolve the NPC talk index with strict bounds checks and return a bounded message/result structure.
- At least one stable bundled town NPC line is decoded and displayed through shell status or a minimal AppKit-owned transcript surface.
- No-NPC, invalid talk index, truncated record, and unsupported scripted entry produce safe explicit results without mutation.
- Dialogue bytes and any legacy control values are not treated as C strings without validated lengths.

## Affected Systems

- Legacy source files: `Sources/UltimaMain.c` `Transact`/`Look`; `Sources/UltimaMisc.c` `Speak` and town interaction helpers
- Portable core: focused location interaction/talk decoding APIs
- Modern shell: command routing and bounded transcript/status presentation
- Modernization harness: bundled talk fixtures and malformed synthetic records
- Documentation: supported talk subset and deferred opcodes

## Validation Plan

Cover one known NPC line, direction/adjacency selection, no NPC, invalid indices, unsupported control entries, length bounds, and no save mutation. Run the standard harness, Swift tests, and boundary scan.

Manual validation: enter the selected town, stand beside the documented NPC, use Talk, verify the expected line, then Talk toward an empty tile and confirm distinct safe feedback.

## Open Questions

- Is legacy Talk invoked through `T`/Transact for this context, or should the modern shell expose a temporary explicit Talk command while preserving command semantics?
- Which smallest talk record subset can be rendered without prematurely implementing `Speak`'s full scripting behavior?
