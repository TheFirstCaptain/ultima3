# F-003A: Portable Core Layout Decision

## Objective

Choose the initial directory structure, naming convention, and API shape for portable core modules used by the harness.

## User Value

A small layout decision prevents ad hoc extraction and gives future porting tasks a stable place to put code that is no longer tied to Carbon, QuickTime, resources, or app startup.

## Non-Goals

- Do not extract game behavior in this feature unless explicitly approved as a proof of concept.
- Do not choose the final modern macOS UI framework.
- Do not reorganize legacy `Sources/` broadly.

## Acceptance Notes

- Recommended portable core location is documented.
- Naming convention for headers and functions is documented.
- Initial guidance for state ownership is documented.
- `DECISION_LOG.md` is updated if the decision is accepted.

## Affected Systems

- Legacy source files: none expected
- Modernization harness: possible include path or Makefile planning
- Resources/assets: none
- Build or tooling: proposed harness build layout
- Documentation: `DECISION_LOG.md`, `ARCHITECTURE.md`, `docs/modernization/porting-strategy.md`

## Validation Plan

Documentation review. If a proof of concept is requested, compile one tiny portable module through `make -C harness test`.

## Open Questions

- Should portable core files avoid all legacy Mac type names, including `Str255`, or temporarily define compatible byte-array aliases?
- Should harness-only shims live beside tests or under the portable core?
