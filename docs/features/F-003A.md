# F-003A: Portable Core Layout Decision

## Objective

Choose the initial directory structure, naming convention, and API shape for portable core modules used by the harness.

## User Value

A small layout decision prevents ad hoc extraction and gives future porting tasks a stable place to put code that is no longer tied to Carbon, QuickTime, resources, or app startup.

## Non-Goals

- Do not extract game behavior in this feature unless explicitly approved as a proof of concept.
- Do not choose the final modern macOS UI framework.
- Do not reorganize legacy `Sources/` broadly.

## Acceptance Notes

- Recommended portable core location is documented.
- Naming convention for headers and functions is documented.
- Initial guidance for state ownership is documented.
- `DECISION_LOG.md` is updated if the decision is accepted.

## Decision

Create the first portable game core under a new top-level `Core/` directory, outside the legacy `Sources/` tree:

```text
Core/
  include/
    u3_<domain>.h
  src/
    u3_<domain>.c
```

Initial modules should be plain C and should be compiled directly by the harness, for example with `-I../Core/include` and explicit `../Core/src/u3_<domain>.c` inputs. Do not add the core to the legacy Xcode target until a later integration task needs it.

Naming convention:

- Files use lowercase snake case with a `u3_` prefix, such as `u3_map.h`, `u3_text.h`, and `u3_combat.h`.
- Public functions use `u3_<domain>_<operation>`, such as `u3_map_get_heading`.
- Public structs and typedefs use `u3_<domain>_<name>`, such as `u3_map_state`.
- Public constants and macros use `U3_<DOMAIN>_<NAME>`.
- Legacy function names may be mentioned in comments and tests, but should not be the portable public API.

State guidance:

- Prefer pure functions for behavior that has no state.
- Where legacy behavior depends on globals, introduce small caller-owned state structs and pass pointers explicitly.
- Use fixed-width standard C types for byte-sensitive behavior.
- Do not expose Classic Mac types from portable headers. If Pascal string behavior needs the exact byte layout, define a portable alias in the relevant core header, for example a `uint8_t` 256-byte type, rather than using `Str255`.
- Harness-only compatibility shims belong in `harness/`; reusable game behavior belongs in `Core/`.

Status: complete as of 2026-05-30. The accepted decision is recorded in `DECISION_LOG.md`.
