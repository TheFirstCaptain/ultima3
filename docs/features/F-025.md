# F-025: Overworld Movement Smoke

## Objective

Allow a minimal party position to move on an overworld map through modern input and redraw the result.

## User Value

This is the first candidate slice where the modern shell starts to feel playable: input changes game state and rendering reflects the change.

## Non-Goals

- Do not port every overworld rule, encounter, town, dungeon, ship, or terrain interaction.
- Do not implement combat transitions.
- Do not add real save timing beyond any smoke persistence explicitly inherited from earlier features.
- Do not bypass the input, render, resource, or persistence adapter boundaries.

## Dependencies

- F-018 provides input event intake.
- F-022 provides resource-backed render data.
- F-023 provides a shell tick or update path.
- F-021 provides a loadable initial game state.

## Acceptance Notes

- A known map fixture and party position can be loaded.
- Arrow or keypad input changes the party position according to a minimal bounded rule.
- The renderer updates visibly after movement.
- Harness coverage validates the movement rule and boundary behavior for the smoke map.
- Manual HIL verification confirms visible movement in the modern shell.

## Implementation Notes

- Added `u3_overworld` as a portable bounded smoke movement contract with cardinal commands, explicit state, and movement result data.
- The smoke starts on the new-game current Sosaria map fixture at position `5,5` within the first `11x11` viewport and bounds movement to that viewport for this slice.
- The smoke renderer builds a resource-backed tile frame from the save-domain `MAPS` record and overlays a party marker at the current smoke position.
- AppKit translates arrow keys plus keypad `8/2/4/6` into portable overworld commands. The existing shell tick consumes one queued input event, applies movement, and redraws the frame.
- Harness coverage validates initialization, cardinal movement, blocked edges, unknown commands, map-backed render output, and fallback behavior.
- Swift shell coverage validates movement through the tick path, visible party-marker relocation in the render frame, and blocked edge reporting.

## Open Questions

- Resolved for this slice: use a bounded fixture viewport, not full legacy overworld wrapping.
- Resolved for this slice: support arrows and keypad `8/2/4/6`; legacy letter commands remain future work.
