# F-025: Overworld Movement Smoke

## Objective

Allow a minimal party position to move on an overworld map through modern input and redraw the result.

## User Value

This is the first candidate slice where the modern shell starts to feel playable: input changes game state and rendering reflects the change.

## Non-Goals

- Do not port every overworld rule, encounter, town, dungeon, ship, or terrain interaction.
- Do not implement combat transitions.
- Do not add real save timing beyond any smoke persistence explicitly inherited from earlier features.
- Do not bypass the input, render, resource, or persistence adapter boundaries.

## Dependencies

- F-018 provides input event intake.
- F-022 provides resource-backed render data.
- F-023 provides a shell tick or update path.
- F-021 provides a loadable initial game state.

## Acceptance Notes

- A known map fixture and party position can be loaded.
- Arrow or keypad input changes the party position according to a minimal bounded rule.
- The renderer updates visibly after movement.
- Harness coverage validates the movement rule and boundary behavior for the smoke map.
- Manual HIL verification confirms visible movement in the modern shell.

## Open Questions

- Should the first smoke use legacy overworld wrapping rules or a bounded fixture map?
- Which command keys should be supported first: arrows, keypad, legacy letter commands, or the already queued raw key events?

## Status

Proposed candidate after F-021 through F-023.
