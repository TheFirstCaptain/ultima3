# F-001: Modernization Analysis Baseline

## Objective

Create the first analysis baseline for modernizing the legacy Ultima III Mac application into a modern Apple silicon-compatible macOS app.

## User Value

This gives maintainers and agents a shared map of the legacy codebase before porting begins, reducing guesswork and avoiding broad rewrites without context.

## Non-Goals

- Do not port code.
- Do not choose the final modern app framework.
- Do not change legacy source or assets.
- Do not create tests or harness executables yet unless a child analysis explicitly identifies them as future work.

## Acceptance Notes

- Child analyses F-001A through F-001D are completed or explicitly deferred.
- `docs/modernization/legacy-inventory.md` is populated with initial findings.
- `docs/modernization/validation-strategy.md` is updated with candidate characterization targets.
- Key open questions are recorded in the relevant docs.

## Analysis Summary

- F-001A mapped source responsibilities across startup, state, map/dungeon, combat/spells, text, graphics, input/events, sound/music, preferences/dialogs, and save/load.
- F-001B identified major Apple silicon blockers: Carbon UI/events, QuickDraw/GWorld/PICT rendering, QuickTime/Sound Manager audio, Resource Manager persistence, 32-bit/PPC build settings, and Pascal string/handle assumptions.
- F-001C identified `UltimaMain.c` as the primary shared-state owner and `UltimaMain.c`, `UltimaMacIF.c`, `UltimaGraphics.c`, `UltimaMisc.c`, `UltimaSpellCombat.c`, and `UltimaDngn.c` as coupling hotspots.
- F-001D recommended map/math accessors in `Sources/UltimaMisc.c` as the first characterization target.

## Execution Model

F-001A through F-001D may be handled by focused subagents to keep context small. Each child analysis should produce concise doc updates, cite inspected files or search terms, and summarize findings back into this umbrella feature before F-001 is marked `Done`.

## Affected Systems

- Legacy source files: `Sources/`
- Modernization harness: `ENGINEERING_HARNESS.md`, `ARCHITECTURE.md`
- Resources/assets: inventory only
- Build or tooling: no changes expected
- Documentation: `docs/modernization/`

## Validation Plan

Validate by documentation review. Confirm each child analysis cites files inspected, states uncertainty explicitly, and avoids source or asset changes.

## Open Questions

- Which harness technology should be used for the first map/math characterization target?
- How should legacy globals and Mac types be shimmed for early tests?
