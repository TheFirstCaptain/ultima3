# F-045: Dungeon Passive Turn Mechanics and Encounter Requests

## Objective

Characterize and add the passive dungeon turn mechanics that run after handled dungeon commands: party aging/status updates, light decay integration, random encounter request generation, and safe combat-screen handoff metadata without starting the full combat loop.

## User Value

Dungeons should feel alive after movement lands. This feature prepares the modern shell for combat while keeping the first encounter slice testable and non-modal.

## Non-Goals

- Do not implement the full `Combat()` loop or player/monster combat command UI.
- Do not mutate combat victory, death, loot, or post-combat map restoration.
- Do not implement traps, fountains, marks, writing, wind, or chest opening.
- Do not bypass the shell tick with a legacy-style modal dungeon loop.

## Dependencies

- F-043 provides handled dungeon command results and surface return.
- F-044 owns light state and turn-based light decay.
- F-015 characterizes representative `CONS` combat-screen resource loading.
- F-009A through F-009C provide bounded combat resolution helpers for later slices.

## Acceptance Notes

- A single portable post-command dungeon turn function accepts explicit state, current tile value, light value, party-size/status inputs, and deterministic random rolls.
- The result reports whether passive aging/status work is requested, whether light was decremented, and whether a random encounter should start.
- Encounter requests include the characterized monster type and combat-screen setup metadata needed by a later combat-entry feature, but do not start combat in this slice.
- Empty floor encounters follow the legacy probability and level-bounded monster selection with deterministic harness coverage.
- Blocked and invalid commands run passive mechanics after the shared turn cost; valid surface exits suppress encounter requests while restoring Sosaria.
- Space/Pass is handled as a dungeon command and runs the passive post-turn path without movement or redraw.

## Affected Systems

- Legacy source files: `Sources/UltimaDngn.c` `dungeonmech`; `Sources/UltimaSpellCombat.c` combat entry references
- Portable core: dungeon post-turn result contracts and deterministic encounter request data
- Modern shell: dungeon tick integration, status reporting, and deferred encounter presentation
- Modernization harness and Swift tests: deterministic passive turn and encounter-request coverage
- Documentation: `ModernShell/README.md`, validation strategy if new characterization targets are added

## Validation Plan

Covered no-encounter and encounter random branches, level-bounded monster type selection, light decay ordering, invalid/blocked command handling, pass handling, exit suppression, and status-only shell reporting. Run `make -C harness test`, `swift test`, and the Core boundary scan.

Manual validation: enter the dungeon, perform repeated movement/pass commands with deterministic or observable status, confirm encounter requests are reported without entering modal combat, and confirm exiting still restores Sosaria and save eligibility.

## Open Questions

- Resolved for this slice: encounter requests remain status-only and allow continued navigation until combat-entry work lands.
- Resolved for this slice: dungeon encounters report the legacy dungeon floor combat screen handoff, `CONS` 402 (`BASERES + 2`), plus marker tile `0x40`.
