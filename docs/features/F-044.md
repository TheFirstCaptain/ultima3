# F-044: Dungeon Light and Torch State

## Objective

Add a bounded dungeon light model to the portable session and shell so Ignite consumes a party torch, sets visible dungeon range, decays on handled dungeon turns, and redraws the perspective view accordingly.

## User Value

Dungeon navigation should stop depending on a diagnostic visibility assumption. Players need a visible, understandable way to manage light before deeper dungeon mechanics such as encounters, traps, and special tiles become playable.

## Non-Goals

- Do not implement spell-based light, time negate, wind, traps, combat encounters, or dungeon special tiles.
- Do not persist while inside a dungeon or add autosave timing.
- Do not redesign the renderer beyond exposing light-dependent visibility to the existing dungeon frame path.
- Do not implement inventory management beyond the torch count mutation needed for Ignite.

## Dependencies

- F-043 wires dungeon commands into the shell tick path and resolves the first visibility policy question.
- F-036 keeps save policy Sosaria-only while dungeon sessions mutate transient state.
- F-024 exposes party and roster state needed to inspect and mutate torch inventory bytes.

## Acceptance Notes

- Dungeon sessions carry an explicit torch/light counter derived from legacy `gTorch` behavior.
- Ignite is accepted only while a dungeon session is active, auto-selects the first active party member with living legacy status (`G` good or `P` poisoned) and a torch as the bounded first-slice equivalent, decrements that member's torch count, and sets light to the characterized legacy value.
- A failed Ignite attempt reports no-torch, invalid-character, or not-in-dungeon feedback without corrupting session or save state.
- The torch inventory decrement is applied to the current in-memory native document, marks it as unsaved, remains unsafe to write while the party is in dungeon mode, and becomes durable only after returning to Sosaria and using the existing manual Save command.
- The modern smoke new-game builder seeds a small torch count for active occupied party members when the source template has none, because shop/equipment acquisition is outside this slice.
- Handled dungeon turns decrement light when it is positive; zero light reports darkness and limits or suppresses perspective geometry according to the selected F-043 policy.
- Redraw, status, and audio feedback are explicit result fields; AppKit remains the shell/input/audio owner.

## Affected Systems

- Legacy source files: `Sources/UltimaMain.c` `Ignite`; `Sources/UltimaDngn.c` `DungeonStart` light checks and turn decay
- Portable core: dungeon/session light state and bounded Ignite result contract
- Modern shell: dungeon command dispatch, setup prompt handling if needed, redraw/status/audio mapping
- Modernization harness and Swift tests: light decay, torch inventory mutation, invalid Ignite paths, and dark render behavior
- Documentation: `ModernShell/README.md`, validation notes, and feature tracker

## Validation Plan

Cover successful Ignite, poisoned living character Ignite, no-torch rejection, invalid character selection, non-dungeon rejection, current-document dirty-state marking, light decay across handled dungeon commands, zero-light rendering/status, save rejection while in dungeon, and return-to-Sosaria save/reload persistence of the decremented torch count. Run `make -C harness test`, `swift test`, and the Core boundary scan.

Manual validation: enter Dungeon Doom, confirm the dungeon starts dark with light `0`, ignite a torch from the first living active party member with torch inventory, move/turn until the light counter changes in status, confirm the frame redraws, confirm Save remains rejected until returning to Sosaria, then save, reload, and confirm the torch count change persisted. Legacy good and poisoned statuses are both treated as living for Ignite.

## Open Questions

- Resolved for this slice: Ignite uses bounded auto-selection of the first active party member with living legacy status (`G` or `P`) and torch inventory. A prompt can be added later when raw numeric character selection is available without conflicting with movement command normalization.
- Resolved for this slice: the status overlay exposes the exact light counter for validation.

## Status

Completed on 2026-07-11. Dungeon sessions now carry explicit light, start dark, Ignite consumes native roster torch inventory and sets light to 255, handled dungeon commands decay positive light, and return-to-Sosaria manual save persists the decremented torch count.
