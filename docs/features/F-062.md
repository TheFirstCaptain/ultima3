# F-062: Representative Destination Session Coverage

## Objective

Generalize resource-backed destination sessions beyond the current LCB Towne and Dungeon Doom slices through a small documented set of representative additional destinations.

## User Value

The modernization shell needs to prove that the session architecture is not hard-coded to one town and one dungeon. Broader destination coverage enables real route testing across Sosaria.

## Non-Goals

- Do not implement all location-specific NPC scripts, shops, quests, or dungeon specials in this feature.
- Do not require every destination family to land in this one tranche; selected representative coverage is enough.
- Do not redesign the renderer or persistence format.
- Do not make transient destination coordinates durable over Sosaria coordinates.
- Do not guarantee every bundled destination works until its resource family has fixture coverage.

## Dependencies

- F-015 characterizes representative town/castle/Ambrosia/dungeon records.
- F-035 through F-039 provide transition, session loading, movement, and return policy.
- F-041 through F-043 provide dungeon session initialization and surface return.
- F-040 and F-057 provide bounded town interaction paths that should remain optional per destination.

## Acceptance Notes

- The location transition/session loader supports a small documented set of additional bundled destinations through data-driven resource ids instead of one-off shell branches.
- The selected set must include at least one additional two-dimensional destination. A second representative destination from Ambrosia or another dungeon is included only if its resource family can be characterized safely within the tranche.
- Each selected destination validates required `MAPS`, `MONS`, and `TLKS` or dungeon record lengths before activation.
- Entry failures leave the current Sosaria document and frame unchanged.
- Two-dimensional destinations support non-wrapping movement and safe exits; selected dungeon entries initialize level, position, heading, and return context correctly.
- Existing LCB Towne and Dungeon Doom tests remain stable.

## Affected Systems

- Legacy source files: location table, map loading, town/castle/Ambrosia/dungeon entry rules
- Portable core: location transition/session metadata and resource validation
- Modern shell/resource adapter: data-driven destination loading and active-session ownership
- Modernization harness and Swift tests: fixture-backed entry, movement, failure, and return coverage
- Documentation: `ModernShell/README.md`, feature tracker, possibly resource inventory notes

## Validation Plan

Cover at least one additional town or castle, optionally one additional dungeon or Ambrosia-style destination if safely characterized, malformed/missing resource failure, exact Sosaria restoration, and save rejection while inside destinations. Run `make -C harness test`, `swift test`, and the Core boundary scan.

Manual validation: walk to each documented destination, enter, move, interact with supported safe commands, exit, save/reload outdoors, and confirm the party returns to the expected Sosaria coordinates.

## Open Questions

- Which representative destinations form the safest tranche: Lord British Castle, one additional town, Ambrosia, or one additional dungeon?
- Should destination metadata stay derived from native `MISC` location-table bytes, or should the adapter cache a validated index for shell diagnostics?
- Are castle-specific commands distinct enough to need their own feature before broad castle coverage?
