# F-042: Portable Dungeon Perspective Render Contract

## Objective

Produce a portable render description for the visible dungeon corridor from explicit dungeon bytes, level, position, and heading, then draw that description in the existing AppKit host.

## User Value

A dungeon session must look like a dungeon rather than a debug status screen. This feature makes the first-person spatial layout visible while preserving renderer/core separation.

## Non-Goals

- Do not port QuickDraw calls or expose AppKit drawing types to Core.
- Do not implement movement, encounters, animation, torch decay, secret discovery, or all decorative effects.
- Do not choose a final GPU renderer.
- Do not modify bundled dungeon artwork.

## Dependencies

- F-041 provides a validated active dungeon session.
- F-008 provides bounded dungeon cell access and heading semantics.
- F-017 provides the portable render-frame and AppKit adapter direction.

## Acceptance Notes

- A pure portable function derives a bounded near-to-far wall/door/ladder/chest description from explicit dungeon state.
- The result uses renderer-neutral commands or a dungeon scene description; it does not embed AppKit, Core Graphics, or asset URLs.
- At minimum, front/left/right walls across the characterized visible depth and the current-cell ladder/chest features are distinguishable.
- Out-of-range state and malformed dungeon buffers return a safe empty/error result.
- The AppKit adapter renders the selected bundled dungeon fixture deterministically at window resize scales.
- Rendering does not mutate dungeon or save state.

## Affected Systems

- Legacy source files: `Sources/UltimaDngn.c` `DrawDungeon`, `DungeonBlock`, wall/door/ladder/chest drawing helpers
- Portable core: `u3_dungeon` plus renderer-neutral dungeon scene/render types
- Modern shell: AppKit render adapter and game host view
- Modernization harness: synthetic corridor geometry and bundled dungeon fixture checks
- Resources/assets: existing dungeon graphics only if the adapter can consume them without conversion

## Validation Plan

Test headings, near/far walls, doors, ladders, chests, boundary cells, malformed buffers, command capacity, and no state mutation. Add deterministic Swift adapter tests where feasible and capture a manual screenshot for review. Run the standard harness, Swift tests, and boundary scan.

Manual validation: enter the selected dungeon and compare several headings/cells with fixture expectations, resize the window, and confirm geometry remains stable and unclipped.

## Open Questions

- Should the first adapter draw vector-like geometry from the scene description or map commands onto the bundled dungeon sprite sheets?
- Which legacy occlusion ordering must be characterized before the scene contract is considered stable?
