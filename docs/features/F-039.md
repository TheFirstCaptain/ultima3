# F-039: First Town Entry and Sosaria Return Round Trip

## Objective

Complete the first real town round trip: accept the F-035 transition, enter the F-037 town session at the legacy initial coordinate, navigate it, and restore the prior Sosaria session when the player exits at a map boundary.

## User Value

This is the first complete multi-map gameplay loop in the modern shell. Players can enter a recognizable place, move within it, and return to the same overworld location.

## Non-Goals

- Do not persist destination map or NPC mutations.
- Do not implement talk, shops, stealing, combat, or town NPC turns.
- Do not add autosave; manual save remains unavailable while inside the town.
- Do not generalize the completed round trip to castles or dungeons until their distinct rules are characterized.

## Dependencies

- F-035 detects explicit town entry.
- F-037 loads and owns the town session and return context.
- F-038 provides 2D destination movement and exit requests.
- F-036 defines dirty-state, quit, and save restrictions.

## Acceptance Notes

- Accepting the selected transition sets the in-memory party location mode to town and uses the legacy initial town coordinate (`x = 1`, `y = 32`, heading east) through an explicit state transition.
- The handled Enter command applies the shared turn cost, so entry marks the in-memory document dirty even though it cannot be saved until Sosaria is restored.
- The pre-entry Sosaria map, position, and creature state remain available for restoration without writing an intermediate durable save.
- Manual save is rejected while the active session is not Sosaria, with a clear status.
- A location edge exit restores location mode zero, the exact pre-entry Sosaria coordinate, Sosaria frame data, and overworld input behavior.
- Failed entry or exit restoration is transactional: it retains a valid current session and save document.
- Quit with an active transient town session follows the F-036 policy without silently persisting an unsafe location state.

## Affected Systems

- Legacy source files: `Sources/UltimaMain.c` `Enter` and `Routine6E6B`; `Sources/UltimaMisc.c` `PushSosaria`/`PullSosaria`
- Portable core: location transition/session state
- Modern shell: current-session routing, save eligibility, quit handling, render refresh
- Modernization harness and Swift tests: transition transaction and round-trip coverage
- Documentation: `ModernShell/README.md` and save-policy notes if behavior is clarified

## Validation Plan

Test entry state, exact return coordinates, save rejection in town, non-persistence of transient destination data, failed-load rollback, boundary exit, and restoration after multiple town moves. Run the standard harness, Swift tests, and boundary scan.

Manual validation: enter the documented town, move several steps, attempt and observe rejected Save, exit at a boundary, confirm the original Sosaria coordinate/frame returns, then save and relaunch successfully outdoors.

## Open Questions

- Should quitting from a transient destination always require returning to Sosaria, or may Discard terminate immediately under the existing dirty-state policy?
- Resolved for this slice: town entry marks the document dirty because the handled Enter command advances the shared move counter; returning to Sosaria does not clear that pending durable change.
