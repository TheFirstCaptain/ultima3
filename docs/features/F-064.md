# F-064: Playability Stabilization and Release Smoke

## Objective

After the preceding tranche features land, harden the modernization shell for longer manual play sessions with command discoverability, save/load resilience, repeatable smoke scripts, and packaging-oriented validation.

## User Value

As more gameplay systems become playable, the project needs a stable way to exercise them without remembering every smoke command or risking accidental save loss.

## Non-Goals

- Do not declare the modern shell feature-complete.
- Do not replace the manual save policy with autosave unless a separate decision accepts it.
- Do not redesign the UI into a final game interface.
- Do not remove legacy project files or historical assets.

## Dependencies

- F-036 provides manual save/load, dirty-state, and quit behavior.
- F-057 through F-063 add enough gameplay breadth to make longer manual sessions worthwhile. F-064 is intentionally last in this tranche; if pulled earlier, its release-smoke scope must shrink to already completed gameplay.
- Existing harness and Swift smoke tests provide the automated baseline.

## Acceptance Notes

- The shell exposes a compact command/help or inspector surface for currently supported gameplay commands without moving game input ownership out of AppKit.
- Save/load error handling and unsaved-change prompts are reviewed against town, dungeon, combat, and Sosaria states.
- A documented release-smoke workflow can create/load a game, move outdoors, enter/exit locations, run the completed dungeon/combat slices, use completed town services, save, reload, and inspect durable state.
- Packaging or launch validation is documented for the SwiftPM app product without implying the legacy Xcode target builds on modern systems.
- Documentation clearly distinguishes playable modern-shell behavior from deferred legacy behavior.

## Affected Systems

- Modern shell: command discoverability, status/help presentation, save/load prompts, and manual smoke affordances
- Tooling: optional smoke scripts or documented command sequences
- Modernization harness and Swift tests: end-to-end smoke coverage where practical
- Documentation: root `README.md`, `ModernShell/README.md`, feature tracker, manual validation notes

## Validation Plan

Cover save/load prompt behavior across Sosaria, town, dungeon, and combat states; command help surface visibility; smoke workflow consistency; and launch/build commands. Run `make -C harness test`, `swift test`, and any packaging or smoke command added by the feature.

Manual validation: follow the documented release-smoke script from new game through movement, town service, dungeon/combat slice, save/reload, and quit prompts.

## Open Questions

- Should discoverability be a status overlay, a menu item, a SwiftUI inspector, or a separate manual document first?
- Should a deterministic debug mode be introduced for manual smoke paths, or should tests remain the only deterministic control surface?
- What minimum packaging check is useful before a full macOS app packaging milestone?
