# F-009: Combat Action Resolution Inventory

## Objective

Map the combat action resolution surface before extracting larger combat behavior.

## User Value

The current portable combat work covers predicates and autocombat decisions, but not the effects of attacks, spells, monster damage, poison, victory, or combat turn progression. This inventory should identify the smallest extractable combat resolution slices and the side effects that must be modeled first.

## Non-Goals

- Do not port the full combat loop in one step.
- Do not implement attack, spell, projectile, poison, victory, or loot resolution during the inventory.
- Do not change existing portable combat APIs unless a narrow documentation correction requires it.
- Do not modify resources or historical assets.

## Acceptance Notes

- Inventory the main combat action functions and side effects in `Sources/UltimaSpellCombat.c`.
- Identify dependencies on globals, rendering, sound, text, random numbers, resources, and party state.
- Recommend the next one to three characterization targets after autocombat and dungeon navigation.
- Capture findings in modernization documentation and, if useful, propose child features for concrete extraction work.
- No code extraction is required for this milestone.

## Affected Systems

- Legacy source files: reference only, primarily `Sources/UltimaSpellCombat.c`, plus any called helpers in `Sources/UltimaMain.c`, `Sources/UltimaMisc.c`, and `Sources/UltimaText.c`
- Modernization harness: none required
- Resources/assets: none
- Build or tooling: none expected
- Documentation: modernization inventory, feature tracker, and possibly new feature docs

## Validation Plan

This is an analysis milestone. Validate by citing inspected files, search terms, and candidate functions. If any harness changes are made incidentally, run:

```bash
make -C harness test
```

## Open Questions

- Which combat side effects can be returned as events from the core instead of executed directly?
- Which behavior depends on random number generation and needs deterministic seed control?
- Should combat resolution be extracted before or after save/resource data is better understood?
