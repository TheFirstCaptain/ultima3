# F-060: Combat Loot and End-State Rewards

## Objective

Add bounded post-victory combat rewards and cleanup so winning combat can award gold, inventory, and status outcomes before restoring the preserved dungeon session.

## User Value

Combat victory currently proves XP and return flow, but it still feels incomplete because loot and reward handling are deferred. This feature makes combat materially useful for party progression.

## Non-Goals

- Do not implement every legacy loot table if a smaller characterized reward slice is safer.
- Do not make transient combat positions, monster records, or combat-screen state durable.
- Do not add town shops or dungeon chest behavior; those stay in their own systems.
- Do not change flee or defeat rewards except to keep their explicit no-loot behavior stable.

## Dependencies

- F-052 provides victory detection, XP award, and source-session restoration.
- F-053 and F-056 provide first inventory mutation patterns.
- F-055 defines flee and defeat exits that should not accidentally receive victory loot.

## Acceptance Notes

- Final-monster defeat can produce bounded reward results for gold and selected inventory categories.
- Reward mutation writes only to the current in-memory native document and is durable only after returning to Sosaria and saving.
- Flee, failed flee, and defeat continue to restore or reject state without victory rewards.
- Reward overflow, full inventory, invalid roster references, and malformed combat setup report explicit statuses without corrupting the document.
- Shell status/audio clearly distinguishes XP, gold, item reward, no reward, and skipped reward cases.

## Affected Systems

- Legacy source files: combat victory, treasure, inventory, and gold reward paths
- Portable core: combat reward contract and native roster mutation helpers where reusable
- Modern shell: victory reward write-back and status/audio reporting
- Modernization harness and Swift tests: deterministic reward, no-reward, overflow, and exit-policy coverage
- Documentation: `ModernShell/README.md`, feature tracker

## Validation Plan

Cover deterministic gold reward, bounded weapon or armour reward if selected, no-loot flee/defeat paths, inventory-full handling, gold cap behavior, document dirty-state marking, and save-after-return durability. Run `make -C harness test`, `swift test`, and the Core boundary scan.

Manual validation: enter dungeon combat, defeat the final monster, confirm reward status, return to Sosaria, save/reload, and inspect party/roster gold and inventory.

## Open Questions

- Should reward ownership live in a combat-specific module or shared party/inventory helpers that town vendors can reuse?
- Which reward class should land first: gold only, food, weapons/armour, or a small mixed table?
- Does combat loot need a short presentation delay from F-058, or is status/audio sufficient for this tranche?
