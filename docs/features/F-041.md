# F-041: Dungeon Entry and Session Integration

## Objective

Extend location transitions to one dungeon entrance, load its 2048-byte dungeon and talk records into an explicit session, and initialize the characterized dungeon position, level, and heading without starting the legacy modal dungeon loop.

## User Value

Dungeons are a major gameplay mode and use a fundamentally different map layout. Loading one through the modern session architecture proves the shell can support more than 2D town maps.

## Non-Goals

- Do not implement perspective rendering, dungeon movement, torches, encounters, traps, chests, fountains, or dungeon talk.
- Do not run `DungeonStart` or call legacy rendering/input code.
- Do not persist an in-dungeon save.
- Do not generalize all seven dungeon fixtures in the first implementation.

## Dependencies

- F-035 establishes explicit Enter transition detection; this feature extends the result family to a characterized dungeon tile.
- F-037 provides session/resource-loading ownership and record validation.
- F-008 provides extracted portable dungeon navigation state conventions.
- F-036 retains Sosaria-only save policy.

## Acceptance Notes

- One stable dungeon entrance coordinate, location index, and resource id are documented from bundled fixtures.
- Explicit Enter produces a dungeon transition request only at the recognized coordinate and dungeon tile class.
- The location session validates exactly 2048 dungeon bytes plus the required talk record before activation.
- Initial state matches the legacy entry contract: level zero, `x = 1`, `y = 1`, and heading one.
- The pre-entry Sosaria session remains restorable and Save is rejected while the dungeon session is active.
- Entry failures leave the active Sosaria session unchanged.

## Affected Systems

- Legacy source files: `Sources/UltimaMain.c` `Enter`; `Sources/UltimaMisc.c` `LoadUltimaMap`; `Sources/UltimaDngn.c` `DungeonStart`
- Portable core: location transition/session state and existing `u3_dungeon` integration
- Modern shell: command dispatch and active-session ownership
- Modernization harness: dungeon transition and bundled resource fixtures
- Documentation: supported dungeon fixture and state contract

## Validation Plan

Cover recognized and unrecognized entry, exact initial state, resource length/type rejection, transaction rollback, active save restriction, and retained return context. Run the standard harness, Swift tests, and boundary scan.

Manual validation: reach the documented dungeon entrance, press Enter, confirm the dungeon session identity and initialized state, and confirm Save is rejected without changing the durable Sosaria document.

## Open Questions

- Resolved: the shell shows the existing diagnostic placeholder and explicit dungeon session status until F-042 provides the perspective render contract.
- Resolved: `TLKS` 412 is loaded and validated now as part of the complete session record family, but dungeon talk remains deferred.
- Resolved: the first supported entrance is Dungeon Doom at Sosaria `(19,57)`, location index `12`, using `MAPS`/`TLKS` resource id `412`. The other six bundled dungeon entrances remain non-enterable in this slice.
