# F-049: Combat Arena Render and Occupant State

## Objective

Render the first transient combat session from resource-backed combat-screen data and explicit party/monster occupant state.

## User Value

Combat should become visible as a combat arena rather than a status message. This creates the first inspectable combat surface before adding command resolution.

## Non-Goals

- Do not implement player or monster turns.
- Do not implement animation, hit flashes, projectiles, or full legacy combat presentation.
- Do not implement combat victory, flee, or return-to-source behavior.
- Do not persist combat-screen state to disk.

## Dependencies

- F-048 creates the transient combat session and loads validated combat-screen data.
- F-015 characterized representative `CONS` combat-screen records.
- F-017 and F-022 provide render-frame and resource-backed render patterns.

## Acceptance Notes

- The combat session exposes a renderer-neutral combat frame from explicit `CONS` bytes and occupant state.
- The first frame distinguishes base terrain, active party slots, and initialized monsters with stable command values or tile ids.
- Invalid occupant coordinates or malformed screen data fall back to explicit safe results without platform UI calls from Core.
- The shell redraws combat frames through the existing AppKit render adapter and reports combat-screen id/session status.
- Combat rendering does not mutate the current native save document or source dungeon/location session.

## Affected Systems

- Legacy source files: `Sources/UltimaSpellCombat.c` combat setup/drawing paths; `Sources/UltimaGraphics.c` combat tile rendering references
- Portable core: combat render/session state contract
- Modern shell: combat frame routing through `GameHostView`
- Resource adapter: copied `CONS` screen bytes and occupant initialization
- Modernization harness and Swift tests: render command contract, malformed coordinates, no-mutation checks
- Documentation: `ModernShell/README.md`, validation strategy if a new render contract is added

## Validation Plan

Cover renderer-neutral frame creation from fixture `CONS` data, party/monster occupant placement, invalid input fallback, no source-buffer mutation, shell redraw/status, and no save mutation. Run `make -C harness test`, `swift test`, and the Core boundary scan.

Manual validation: enter a seeded combat session, confirm the arena renders distinctly from dungeon/town/Sosaria frames, resize the window, and confirm status reports combat session metadata.

## Open Questions

- Resolved for this slice: monster initialization is deterministic and fixture-backed. Exact legacy random monster-count and HP setup remains deferred to later combat state/turn slices.
- Resolved for this slice: combat render overlays reserve `0xE0...0xE7` for monster slots and `0xF0...0xF3` for party slots.

## Status

Completed on 2026-07-13. Added a portable `u3_combat_render` contract that initializes combat terrain/occupants from explicit `CONS` bytes and produces renderer-neutral frames with deterministic monster and party overlays. The modern shell now redraws active combat sessions with that frame while keeping commands, turns, victory, loot, and combat return deferred.
