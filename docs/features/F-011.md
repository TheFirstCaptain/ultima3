# F-011: Modern App Shell Decision Spike

## Objective

Evaluate and choose the first modern macOS app shell direction for the port.

## User Value

The portable core needs a host eventually. A focused decision spike should compare practical shell options against the needs exposed by the extracted core, rendering/audio/resource constraints, Apple silicon support, and maintainability.

## Non-Goals

- Do not build the modern app shell during this spike.
- Do not choose final renderer, audio, persistence, or UI architecture beyond what is necessary to select a shell direction.
- Do not remove or rewrite the legacy Xcode project.
- Do not make asset conversion decisions unless they directly affect shell feasibility.

## Acceptance Notes

- Compare candidate shell approaches such as AppKit, SwiftUI, SDL, or a hybrid approach.
- Evaluate each candidate against input handling, 2D rendering, audio, bundle resources, save files, windowing, menus/preferences, C core integration, and Apple silicon build support.
- Record an accepted or deferred decision in `DECISION_LOG.md`.
- If accepted, propose follow-up feature milestones for a minimal shell, renderer adapter, input adapter, audio adapter, and resource/persistence adapter.

## Child Features

- `F-011A: Modern App Shell Options Matrix` creates the tiered comparison table used as input to this decision spike.

## Decision

Accepted 2026-06-05: use an `AppKit plus SwiftUI hybrid` as the first modern macOS app shell direction.

AppKit owns the application lifecycle, primary windowing, menus, command routing, keyboard and mouse event routing, fullscreen/window behavior, game host view, and concrete save-file location provider. SwiftUI is allowed only for preferences, inspectors, setup flows, and future non-game panels unless a later decision explicitly changes ownership.

This decision chooses the first app shell direction only. It does not choose a final renderer, final audio implementation, final persistence format beyond F-014, or asset conversion path.

## Boundary Rules

- AppKit must own shell lifecycle, primary windows, menus, command routing, keyboard and mouse event intake, fullscreen/window behavior, game host view, and concrete save-file location provider.
- SwiftUI must be limited to preferences, inspectors, setup flows, and future non-game panels unless a later decision explicitly changes ownership.
- SwiftUI must not own the shell lifecycle, command model, primary game surface, deterministic game loop, or first-hop game input routing in the first shell milestone.
- `Core/` must remain free of AppKit, SwiftUI, SDL, Metal, AVFoundation, Foundation filesystem, sandbox, and other platform framework types in public headers.
- Renderer, input, audio, resource, and persistence implementations must remain behind adapters and communicate with the portable core through portable C data, result structs, events, command buffers, or explicit state transitions.
- The shell may provide concrete bundle and save-file URLs or paths, but resource parsing and persistence serialization remain adapter-owned.

## Rationale

The project target is a modern Apple silicon macOS app around a portable C game core. The strongest local evidence points toward a native shell: the legacy project already has Objective-C/Cocoa bridge code, Mac command/menu expectations, preferences/dialog code, and Mac-specific persistence location responsibilities. F-014 also requires the shell to provide concrete save-file locations while keeping persistence serialization behind an adapter.

`AppKit plus SwiftUI hybrid` scored best in F-011A for this repository shape because it preserves native Mac behavior and direct C integration while leaving room for modern SwiftUI preferences and secondary UI. It avoids making SwiftUI the first owner of deterministic game-loop, keyboard-command, and custom-rendering concerns. It also avoids making SDL the primary app shell before cross-platform portability is a first-order requirement.

SDL remains a fallback for renderer/input portability, especially as `SDL3 inside AppKit`, but it is not the first shell direction. A pure SDL shell is rejected for the first milestone because it would weaken native menus, preferences, save-file presentation, accessibility, and app conventions that are central to the accepted macOS target.

## Validation Result

Completed 2026-06-05. The decision was validated against F-011A, `MODERN_APP_SHELL_TECH_COMPARISON.md`, local platform-facing source evidence, the accepted architecture and decision-log boundaries, F-014 persistence ownership, F-012 through F-015 resource/fixture evidence, and current official Apple and SDL documentation.

External documentation notes: Apple documents SwiftUI as cross-Apple-platform UI that can be adopted incrementally alongside AppKit, AVFoundation as a current Apple audio/video framework, and Apple provides current documentation for AppKit, Metal, and universal macOS binaries. SDL documents SDL3 macOS support and exposes Metal renderer integration points, but its own platform page notes maintainers do not test every listed platform.

## Resolved Questions

- The first shell should prioritize native macOS conventions above cross-platform potential.
- The original menu/preferences model should be preserved as a shell-level requirement, not pushed into the portable core.
- Rendering remains an adapter decision after shell selection. The first renderer should be hosted by the AppKit shell, with SDL/Metal still available for later renderer spikes if evidence supports them.
