# F-011A: Modern App Shell Options Matrix

## Objective

Create a comparison matrix for candidate modern app shell approaches before F-011 records a final shell direction.

## User Value

The app shell decision has several competing pressures: native macOS behavior, game-style rendering and input, C core integration, resource/persistence adapters, audio replacement, and long-term maintainability. A shared matrix makes those tradeoffs explicit before choosing the first shell.

## Non-Goals

- Do not choose the final F-011 shell direction in this child feature.
- Do not build a prototype shell.
- Do not choose final renderer, audio, persistence, or UI frameworks beyond shell feasibility scoring.
- Do not remove or modify the legacy Xcode project.

## Tier Scale

| Tier | Meaning | Decision Use |
| --- | --- | --- |
| T4 | Strong fit | Meets the need directly with low expected adapter risk. |
| T3 | Good fit | Meets the need, with manageable integration or implementation work. |
| T2 | Usable with caveats | Feasible, but likely needs wrappers, custom work, or careful constraints. |
| T1 | Weak fit | Possible, but risks awkward integration, behavior drift, or high maintenance. |
| T0 | Poor fit | Does not meet the need without major extra architecture or a separate subsystem. |

## Candidate Shells

| Candidate | Description |
| --- | --- |
| AppKit-first native shell | Native macOS application using AppKit for windowing, menus, commands, event handling, preferences surfaces, and host views. Rendering/audio/resource/persistence stay behind adapters. |
| SwiftUI-first native shell | SwiftUI app lifecycle and views first, with lower-level host views or wrappers only where needed. |
| AppKit plus SwiftUI hybrid | AppKit owns windowing, menus, command routing, game view hosting, and app lifecycle. SwiftUI is used selectively for preferences, inspectors, and future non-game panels. |
| SDL3 shell | SDL owns windowing, input, renderer/audio abstraction, and event loop. Native macOS integration is added only where required. |
| SDL3 inside AppKit hybrid | AppKit owns macOS app shell behavior while an SDL-backed view/window owns the game surface and low-level input/rendering. |

## Option Notes

### AppKit-First Native Shell

AppKit is the strongest native fit for the existing codebase shape. The legacy app already has Objective-C bridge code, Cocoa preference/dialog pieces, menu concepts, window behavior, and Mac-specific user expectations. AppKit also keeps the future save-file provider, menu commands, full-screen behavior, and C integration straightforward.

Main risk: AppKit does not solve game rendering or audio by itself. It still needs deliberate renderer and audio adapters, likely starting with a custom view backed by Core Graphics, Metal, or an intermediate bitmap surface.

### SwiftUI-First Native Shell

SwiftUI is attractive for modern app surfaces and preferences, and Apple documents it as a cross-platform UI framework that can work with AppKit. It is a weaker first owner for the game loop, dense keyboard command handling, custom rendering surface, and legacy command model.

Main risk: a SwiftUI-first app may push core game concerns through wrappers too early. It is better suited to secondary UI once the game host view and adapter contracts are clearer.

### AppKit Plus SwiftUI Hybrid

This is the strongest qualitative native fit in the matrix. AppKit owns the app lifecycle, menus, game window, event routing, save-file provider, and host view. SwiftUI can be adopted incrementally for preferences and other non-game panels. That matches the modernization pattern already used elsewhere: keep core behavior explicit, isolate adapters, and avoid broad rewrites before boundaries are stable.

Main risk: hybrid apps need a clear ownership rule so SwiftUI does not become a second app shell. The initial rule should be AppKit owns shell and game host; SwiftUI owns optional panels only.

### SDL3 Shell

SDL3 is the strongest game-loop and cross-platform candidate. It provides low-level access to input, audio, and graphics hardware, and macOS is listed among platforms SDL3 has been known to work on. It is attractive if the project wants portability beyond macOS.

Main risk: F-011's current target is a modern macOS app, not a cross-platform game port. A pure SDL shell weakens native menus, preferences, save-file presentation, accessibility, and app conventions unless those are rebuilt or bridged.

### SDL3 Inside AppKit Hybrid

This preserves more native shell behavior while using SDL for the game surface. It can be a strong option if renderer/input portability becomes more important than keeping all game-surface code native.

Main risk: two event and lifecycle systems can become awkward. This should be considered only if F-011 decides the game surface benefits enough from SDL to justify hybrid lifecycle complexity.

## Preliminary Read

F-011 should treat `AppKit plus SwiftUI hybrid` and `AppKit-first native shell` as the leading native candidates. `SDL3 inside AppKit` is the leading portability-oriented fallback. `SwiftUI-first` and pure `SDL3 shell` remain viable only if F-011 changes the priority toward declarative UI or cross-platform game portability.

## Evidence Notes

- Local project evidence: `Sources/UltimaMacIF.c`, `Sources/CocoaBridge.m`, `Sources/PrefsDialog.m`, and `Sources/UltimaNew.c` already contain Mac shell, Cocoa bridge, menu, preference, and dialog ownership patterns.
- Local adapter evidence: F-014 defers concrete save-file location to the app shell, so the shell needs a clean storage-provider role.
- Local resource evidence: F-012 through F-015 keep resource and fixture loading portable and byte-oriented, so the shell does not need to own legacy Resource Manager behavior.
- Apple documents AppKit as the macOS application framework, SwiftUI as a UI framework that can work alongside AppKit, Metal as the low-level GPU framework, AVFoundation as the modern media framework, and universal macOS binaries as the path for Apple silicon and Intel slices.
- SDL documents SDL3 as a cross-platform library for low-level access to audio, keyboard, mouse, and graphics hardware, with macOS listed among platforms SDL3 has been known to work on; SDL3 also exposes Metal renderer integration points.
