# F-021: New Game Save Domain Smoke

## Objective

Create and load a minimal modern new-game save-domain state without using legacy UI flows.

## User Value

The modern shell needs a deterministic starting state before it can offer real play, movement, or party setup workflows.

## Non-Goals

- Do not implement the full legacy character creation flow.
- Do not add gameplay save writes beyond the smoke state needed for validation.
- Do not expose AppKit, SwiftUI, or Foundation filesystem types through `Core/`.
- Do not promise legacy save export compatibility.

## Dependencies

- F-020C defines the native save fixture shape.
- F-020D proves save-path and atomic write/read ownership.
- F-014 remains the persistence-format decision source of record.

## Acceptance Notes

- A deterministic new-game native save document can be created from known fixture inputs.
- A portable save-domain loader can validate and expose the required byte domains from a native `u3_save_document`.
- The loaded state preserves the F-014 byte domains: active party, roster, current Sosaria map, current Sosaria creature data, and `MISC` 500..505 tables.
- The shell can trigger and inspect the smoke state without owning serialization details.
- Harness validation covers fixture creation, document load, required-record failure, malformed-record failure, and shell-level smoke tests cover the debug command status.

## Planned State Contract

F-021 should add a small portable loaded-state view in `Core/`, likely in `u3_save` unless the implementation gets large enough to justify `u3_save_state`.

The first state should remain byte-preserving and caller-owned:

```c
typedef struct u3_save_domain_state {
    const uint8_t *party;
    uint32_t party_length;
    const uint8_t *roster;
    uint32_t roster_length;
    const uint8_t *current_sosaria_map;
    uint32_t current_sosaria_map_length;
    const uint8_t *current_sosaria_creatures;
    uint32_t current_sosaria_creatures_length;
    const uint8_t *misc[U3_SAVE_MISC_TABLE_COUNT];
    uint32_t misc_length[U3_SAVE_MISC_TABLE_COUNT];
} u3_save_domain_state;
```

The state should point into the caller-owned native save document bytes, matching the existing `u3_resource` and `u3_save_document` lifetime pattern. It should not allocate, copy, mutate, or expose platform types.

The load API should be explicit and fail closed:

```c
int u3_save_load_domain_state(const u3_save_document *document, u3_save_domain_state *state);
```

Required record checks should include exact type, ID, and byte length:

| Domain | Native record | Required length |
| --- | --- | ---: |
| Active party | `PRTY` 400 | 64 |
| Roster | `ROST` 400 | 1280 |
| Current Sosaria map | `MAPS` 419 | 4101 |
| Current Sosaria creatures | `MONS` 419 | 256 |
| Mutable tables | `MISC` 500..505 | 16, 11, 11, 11, 64, 16 |

The first smoke state should use F-020C's legacy-compatible default party/roster/map layout and zero-filled current Sosaria creatures. Do not introduce placeholder or empty-party semantics for this slice.

## Implementation Plan

1. Add the portable `u3_save_domain_state` loader API.
2. Add harness tests that build the F-020C native new-game fixture, open it with `u3_save_open`, and load the domain state.
3. Assert required domain pointers, exact lengths, byte equality with the source templates, and zero-filled current Sosaria creature bytes.
4. Add negative harness cases for missing required records and wrong required lengths. Use mutated native document bytes rather than shell filesystem paths.
5. Add a shell debug/menu smoke, tentatively Game > Load New Game Smoke, that builds the native new-game smoke document through `ShellResourceAdapter`, loads it through the portable domain loader, and reports concise domain status.
6. Add Swift tests for the shell smoke helper if the implementation is factored outside `AppDelegate`; otherwise rely on manual HIL plus C harness coverage for the loader.
7. Update docs/tracker and run the standard validation set.
