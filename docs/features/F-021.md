# F-021: New Game Save Domain Smoke

## Objective

Create and load a minimal modern new-game save-domain state without using legacy UI flows.

## User Value

The modern shell needs a deterministic starting state before it can offer real play, movement, or party setup workflows.

## Non-Goals

- Do not implement the full legacy character creation flow.
- Do not add gameplay save writes beyond the smoke state needed for validation.
- Do not expose AppKit, SwiftUI, or Foundation filesystem types through `Core/`.
- Do not promise legacy save export compatibility.

## Dependencies

- F-020C should define the native save fixture shape.
- F-020D should prove save-path and atomic write/read ownership.
- F-014 remains the persistence-format decision source of record.

## Acceptance Notes

- A deterministic new-game domain state can be created from known fixture inputs.
- The state can be serialized and loaded through the persistence adapter boundary.
- The shell can trigger or inspect the smoke state without owning serialization details.
- Harness validation covers the fixture shape and round-trip behavior.

## Open Questions

- Should the first smoke state include a placeholder party, an empty roster, or a fully legacy-compatible default party layout?
- Should the shell expose this as a debug command, a temporary menu item, or only harness behavior at first?

## Status

Proposed candidate after F-020 resource and persistence slices settle.
