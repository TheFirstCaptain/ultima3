# F-020: Resource and Persistence Shell Integration

## Objective

Wire shell-owned bundle and save-file locations into the resource and persistence adapters defined by earlier modernization work.

## User Value

The modern app shell must provide concrete locations for bundled data and player saves while preserving F-014's adapter boundary and legacy save timing constraints.

## Non-Goals

- Do not change the F-014 persistence format decision.
- Do not promise exact legacy save export.
- Do not make the persistence adapter depend directly on AppKit, sandbox, Preferences-folder, or application-support APIs.
- Do not convert all resources during this milestone.

## Acceptance Notes

- The AppKit shell provides concrete bundle and save-file URLs or paths to adapters.
- Persistence serialization remains adapter-owned and independent of AppKit types.
- Manual save eligibility and transition/auto-save timing remain aligned with F-014.
- Resource loading can use F-012 through F-015 fixtures or converted bundle data without relying on writable Resource Manager files.

## Planning Result

F-020 should remain an umbrella milestone. The implementation should be split into small child features because resource location, bundled resource reads, save-document creation, shell save-path ownership, and legacy roster import each carry different validation risks.

The first implementation slice should prove location ownership and adapter boundaries before writing gameplay save data. The AppKit shell may resolve concrete paths or URLs, but adapters should receive portable strings or byte buffers and should not receive AppKit, SwiftUI, Foundation URL, sandbox, or Preferences-folder APIs through `Core/`.

## Candidate Child Features

| ID | Candidate | Purpose | Why first/not first |
| --- | --- | --- | --- |
| F-020A | Shell Location Provider Smoke | Let the modern shell report concrete resource root and save-document path candidates without parsing or writing save data. | Best first slice because it verifies shell ownership and HIL visibility with low persistence risk. |
| F-020B | Bundle Resource Adapter Smoke | Use shell-provided resource root to read known bundled bytes through an adapter boundary and validate a known resource fixture. | Good second slice once path ownership is visible. It can reuse F-012/F-013 parser evidence without touching saves. |
| F-020C | Native Save Fixture Builder | Define and generate a synthetic new-game modern save fixture from default templates and zero-filled mutable Sosaria monsters. | Needed before real save writes; should wait until resource adapter input shape is stable. |
| F-020D | Save Path and Atomic Write Smoke | Have the shell provide a save-document path and have an adapter write/read a small validated smoke document atomically. | Useful before gameplay persistence, but should not contain real party/roster state yet. |
| F-020E | Legacy Roster Import Fixture Spike | Add a path for importing a real or synthesized mutable `Ultima III Roster` resource file into the modern save-domain model. | Higher risk and should wait until native save fixture and resource adapter boundaries are proven. |

## Recommended Sequence

1. F-020A, to verify shell-owned path resolution and manual visibility.
2. F-020B, to prove adapters can consume shell-provided resource locations and existing fixture data.
3. F-020C, to establish the byte-preserving native save fixture shape.
4. F-020D, to test save-path ownership and atomic adapter writes without gameplay state.
5. F-020E, to approach compatibility import after the native format and fixture path are clear.

## Affected Systems

- Legacy source files: reference only, primarily `Sources/UltimaMisc.c`, `Sources/UltimaMain.c`, and save/resource startup paths.
- Modern shell: future shell-owned location provider for resource root and save-document paths.
- Portable core: no AppKit, SwiftUI, Foundation URL, or filesystem APIs should be added.
- Modernization harness: future resource/persistence adapter tests using F-012 through F-015 fixture evidence.
- Resources/assets: read-only bundled resources first; save writes should target generated modern documents only.
- Documentation: this umbrella plan and child feature candidates.

## Validation Plan

Use fixture-backed adapter tests for resource and persistence behavior. Add shell-level smoke checks once the AppKit shell can provide real bundle and save locations.

For planning-only work, validation is document review against F-012, F-013, F-014, F-015, `BOUNDARIES.md`, and `DECISION_LOG.md`.

## Open Questions

- Should the first shell save path be a repo-local development path, an Application Support path, or both behind a development/runtime provider distinction?
- Should F-020B consume `MainResources.rsrc` directly through the existing parser or start with already converted fixture files?
- What exact on-disk container should the first native save fixture use: binary chunked document, JSON plus byte blobs, or another simple canonical format?
- Do we have or want to synthesize a mutable legacy `Ultima III Roster` fixture before F-020E?
