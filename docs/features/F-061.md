# F-061: Town Economy and Vendor Services

## Objective

Extend town gameplay from the first healer service into two bounded practical vendor service shapes, selected from supplies such as food/torches and equipment such as weapons/armour.

## User Value

Players need towns to support the resource loop required by dungeon exploration and combat. Buying supplies also exercises save-domain inventory and gold mutation outside combat.

## Non-Goals

- Do not implement every shop, pub, oracle, guild, bribe, or scripted town branch.
- Do not require final tile/NPC service discovery if explicit smoke commands remain safer for the first vendor tranche.
- Do not add new asset formats or a final shop UI panel.
- Do not make town-session coordinates durable over Sosaria return coordinates.

## Dependencies

- F-039 provides town entry and save-safe return.
- F-057 provides the first town service command, prompt, mutation, and save-policy pattern.
- F-056 and F-060 may provide reusable inventory mutation helpers.
- F-024 exposes food, gold, and basic roster summary fields for inspection.

## Acceptance Notes

- At least two additional vendor service shapes are supported, selected for low-risk validation: one stackable supply purchase and one equipment or inventory purchase. The remaining vendor categories stay deferred with explicit unsupported status.
- Services validate active town sessions, selected character or party target, price, available gold, caps, inventory bounds, and cancellation before mutation.
- Successful purchases write to the current in-memory native document, mark it unsaved, and become durable only after returning to Sosaria and saving.
- Insufficient gold, full inventory, invalid target, unsupported service, and cancellation report explicit no-mutation status.
- Existing healer behavior remains stable.

## Affected Systems

- Legacy source files: town shop, food, equipment, and inventory mutation paths
- Portable core: town service/vendor contract and reusable inventory/gold helpers
- Modern shell: command routing, prompt handling, status/audio feedback, and document write-back
- Modernization harness and Swift tests: deterministic purchase, rejection, cancel, and save-durability coverage
- Documentation: `ModernShell/README.md`, feature tracker

## Validation Plan

Cover successful supply purchase, successful equipment or inventory purchase, insufficient gold, cap/full inventory, cancellation, invalid active character, document dirty-state marking, town save rejection, and save/reload after returning to Sosaria. Run `make -C harness test`, `swift test`, and the Core boundary scan.

Manual validation: enter LCB Towne or another selected town, buy supplies and equipment, inspect roster state, return to Sosaria, save/reload, and confirm persistence.

## Open Questions

- Should the next services be explicit smoke commands or discovered through adjacent NPC/shop tiles?
- Should gold spend come from the selected roster member, the leader, or a party-level policy matching characterized legacy behavior?
- Which inventory fields need clearer shared helpers before adding multiple vendors?
