# F-057: First Town Service Slice

## Objective

Add the first bounded town service interaction, such as healing or a shop, using validated town session resources and in-memory native document mutations.

## User Value

Town sessions should start supporting useful non-navigation gameplay. A first service loop gives players a reason to enter towns beyond movement and static NPC talk.

## Non-Goals

- Do not implement every town service, shop type, pub branch, or dialogue tree.
- Do not implement full economy balancing beyond the selected service.
- Do not implement new UI panels if the interaction can use existing keyboard-prompt/status patterns.
- Do not make town-session coordinates durable over Sosaria return coordinates.

## Dependencies

- F-039 provides town entry and Sosaria return.
- F-040 provides bounded town NPC lookup and text display.
- F-036 defines manual save durability.
- F-024/F-030 expose party/roster data in the current native document.

## Acceptance Notes

- A selected first town service is recognized from validated town session state or command context.
- The service uses non-blocking input for any needed character/item confirmation.
- Gold, HP/status, food, or inventory mutations selected for the service write to the current in-memory native document and mark it unsaved.
- Invalid service state, insufficient resources, unsupported service entries, and cancellation report explicit status without corrupting town or save state.
- Save remains rejected while in town mode and becomes available after returning to Sosaria under the existing policy.

## Affected Systems

- Legacy source files: likely `Sources/UltimaTown.c`-equivalent behavior inside `Sources/UltimaMain.c`, `Sources/UltimaText.c`, and town interaction helpers
- Portable core: first town service contract
- Modern shell: town command routing and prompt handling
- Save adapter: selected roster/party/economy mutation
- Modernization harness and Swift tests: valid service, cancellation, insufficient resource, mutation durability
- Documentation: `ModernShell/README.md`, feature tracker

## Validation Plan

Cover valid service mutation, invalid service state, cancellation, insufficient resource, status/audio/redraw if applicable, dirty-state marking, town save rejection, and save/reload after returning to Sosaria. Run `make -C harness test`, `swift test`, and the Core boundary scan.

Manual validation: enter LCB Towne, invoke the selected service, exercise success and failure cases, return to Sosaria, save, reload, and inspect party/roster state.

## Open Questions

- Which service should be first: healer, food, weapons/armour, or another low-risk interaction?
- Should service discovery come from map tile adjacency, NPC talk entries, or an explicit smoke command for the first slice?
