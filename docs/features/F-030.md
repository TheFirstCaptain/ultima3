# F-030: Current Native Save Load and Write

## Objective

Replace the current save smoke path with a shell-held native save document that can be loaded, mutated by setup flows, and atomically written back through the persistence adapter.

## User Value

Modern character and party setup needs to survive app restarts. This feature moves persistence from fixture smoke testing toward a real save/load workflow while keeping the native save format bounded and testable.

## Non-Goals

- Do not implement exact classic roster export.
- Do not add auto-save timing for all legacy transitions.
- Do not save while the party is away from Sosaria unless an explicit transition path has been characterized.
- Do not add file picker or document-browser polish unless required for a functional first path.

## Dependencies

- F-014 defines native save with legacy import direction.
- F-020D provides atomic write/read smoke behavior.
- F-021 loads save-domain state.
- F-030 should land before shell-integrated F-028 and F-029 mutation paths so those features have one shared current-save owner. F-028 and F-029 can still characterize portable byte mutation helpers independently before shell integration.

## Acceptance Notes

- The shell can hold a current native save document instead of rebuilding a fresh fixture for each smoke command.
- Loading a new-game fixture initializes current shell save state.
- Setup-flow mutations can target the current save state in later features; this slice establishes the shared owner and document-based inspection path without adding mutation APIs.
- Saving atomically writes the current native document through `ShellSaveAdapter`.
- Manual save honors the legacy outdoor-Sosaria restriction for durable writes.
- Failed validation or write attempts preserve the previous valid document.

## Implementation Notes

- `ShellSmokeState` now owns an optional current native save document as `Data`.
- `Game > Load New Game Smoke` builds the native new-game document once, stores it as the current save, refreshes the overworld frame from that document, and reports current domain lengths.
- `Game > Write Save Smoke` writes the current document through `ShellSaveAdapter`; it no longer rebuilds a fresh fixture at save time.
- `Game > Load Saved Smoke` reads the Application Support save candidate through `ShellSaveAdapter`, validates it as a complete save-domain document, stores it as the current save, and refreshes shell inspection/render state.
- `Game > Inspect Party/Roster` prefers the current save document when one is loaded, falling back to the fixture-backed path only when no current document exists.
- `ShellResourceAdapter` can now describe, inspect, and render from caller-provided native save document bytes.
- `ShellSaveAdapter` exposes validated document readback for shell-owned save locations.

Manual validation:

1. Run `swift run Ultima3ModernShell`.
2. Choose `Game > Write Save Smoke` before loading a current document and confirm the status reports `Save Missing current document`.
3. Choose `Game > Load New Game Smoke` and confirm the status reports `Current Save OK party 64 roster 1280 map 4101 creatures 256 misc 16/11/11/11/64/16`.
4. Choose `Game > Inspect Party/Roster` and confirm the default party readout is shown from the current document.
5. Choose `Game > Write Save Smoke` and confirm the status reports `Save OK Smoke | Save Read OK Smoke`.
6. Quit and relaunch the shell.
7. Choose `Game > Load Saved Smoke`, then `Game > Inspect Party/Roster`, and confirm the saved current document loads and inspects successfully.
