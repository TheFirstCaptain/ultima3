# F-030: Current Native Save Load and Write

## Objective

Replace the current save smoke path with a shell-held native save document that can be loaded, mutated by setup flows, and atomically written back through the persistence adapter.

## User Value

Modern character and party setup needs to survive app restarts. This feature moves persistence from fixture smoke testing toward a real save/load workflow while keeping the native save format bounded and testable.

## Non-Goals

- Do not implement exact classic roster export.
- Do not add auto-save timing for all legacy transitions.
- Do not save while the party is away from Sosaria unless an explicit transition path has been characterized.
- Do not add file picker or document-browser polish unless required for a functional first path.

## Dependencies

- F-014 defines native save with legacy import direction.
- F-020D provides atomic write/read smoke behavior.
- F-021 loads save-domain state.
- F-030 should land before shell-integrated F-028 and F-029 mutation paths so those features have one shared current-save owner. F-028 and F-029 can still characterize portable byte mutation helpers independently before shell integration.

## Acceptance Notes

- The shell can hold a current native save document instead of rebuilding a fresh fixture for each smoke command.
- Loading a new-game fixture initializes current shell save state.
- Setup-flow mutations update the current save state in memory.
- Saving atomically writes the current native document through `ShellSaveAdapter`.
- Manual save honors the legacy outdoor-Sosaria restriction for durable writes.
- Failed validation or write attempts preserve the previous valid document.
