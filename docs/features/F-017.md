# F-017: Renderer Adapter Spike

## Objective

Define the first renderer adapter contract for drawing Ultima III game output inside the AppKit-owned shell chosen by F-011.

## User Value

The modern shell needs a way to display maps, combat screens, text, cursors, and UI panels without letting rendering APIs leak into the portable core.

## Non-Goals

- Do not port every legacy drawing path.
- Do not convert all assets.
- Do not require the final renderer to be Core Graphics, Metal, SDL, or another specific backend until evidence supports it.
- Do not add rendering dependencies to `Core/`.

## Acceptance Notes

- First prototype backend is `AppKitRenderAdapter`, an AppKit/Core Graphics style command consumer hosted inside `GameHostView`.
- Added `Core/include/u3_render.h` and `Core/src/u3_render.c` as a renderer-independent frame contract using plain C data only.
- The contract models a logical 40x24 frame with bounded clear, rectangle, and tile commands. It includes a helper for 11x11 tile grids, matching the 121-tile combat fixture boundary characterized by F-015.
- The shell renders `u3_render_make_synthetic_tile_frame()` through AppKit, giving the modern host a synthetic visual smoke path without converting legacy tiles yet.
- Metal, SDL, bitmap-backed assets, and final renderer decisions remain open because the shell consumes portable commands rather than game or platform state directly.

## Affected Systems

- Legacy source files: reference only, primarily `Sources/UltimaGraphics.c` and `Sources/UltimaGraphics.h`.
- Portable core: added `u3_render` command/frame contract.
- Modern shell: added `AppKitRenderAdapter` and changed `GameHostView` to consume a portable synthetic frame.
- Modernization harness: added `harness/render_contract_tests.c` and integrated it into `harness/Makefile`.
- Resources/assets: none.
- Documentation: updated this feature record and the modern shell README.

## Validation Plan

Validate with a focused prototype or design note, plus visual/manual checks if a drawable shell exists. Keep harness tests focused on renderer-independent data contracts where possible.
