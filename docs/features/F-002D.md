# F-002D: Combat Predicate Characterization

## Objective

Characterize low-coupling combat movement predicates before broader combat or autocombat work.

## User Value

Combat movement and collision behavior must survive the port. These predicates provide a bounded entry point into combat rules.

## Non-Goals

- Do not characterize the full combat loop.
- Do not port rendering, sound, projectiles, victory, or monster AI.
- Do not include `ValidMove` until sound effects are stubbed or isolated.

## Acceptance Notes

- Characterization covers `CombatValidMove`, `CombatMonsterHere`, and `CombatCharacterHere`, or documents deferrals.
- Tests populate synthetic combat globals and assert exact return codes and lookup priority.
- Monster type/passability cases and reverse-iteration collision behavior are covered.

## Affected Systems

- Legacy source files: `Sources/UltimaSpellCombat.c`, related globals from `Sources/UltimaMain.c`
- Modernization harness: combat predicate tests
- Resources/assets: none expected
- Build or tooling: based on F-002A
- Documentation: validation strategy

## Validation Plan

Run the harness command established by F-002A. Confirm no test requires rendering, audio, resource files, or app startup.

## Open Questions

- Which monster type values are required for first-pass passability coverage?
- Should `ValidMove` be deferred until an audio shim exists?
