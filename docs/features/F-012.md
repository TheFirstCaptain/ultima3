# F-012: Save Resource Fixture Extractor

## Objective

Create a read-only fixture extraction path for the legacy roster/save resource data needed by future persistence characterization.

## User Value

Save compatibility is hard to reason about from source inspection alone. A fixture extractor gives maintainers stable examples of `PRTY`, `ROST`, current Sosaria `MAPS`, current Sosaria `MONS`, and mutable `MISC` data before a modern save adapter is designed.

## Non-Goals

- Do not implement a new save format.
- Do not write or migrate user save files.
- Do not replace Resource Manager loading in the app.
- Do not require final compatibility policy decisions.

## Acceptance Notes

- Identify the minimum fixture data needed for party, roster, current Sosaria map, current Sosaria monsters, and mutable `MISC` resources.
- Choose a read-only extraction approach for legacy roster resource files, using a tool, parser, or documented manual fixture path.
- Preserve byte-level data in a form suitable for harness tests or future adapter tests.
- Document limitations, especially around resource forks and modern filesystem behavior.

## Affected Systems

- Legacy source files: reference only, primarily `Sources/UltimaMisc.c` and `Sources/UltimaMain.c`
- Modernization harness: likely fixture loading or fixture validation support
- Resources/assets: possible read-only sample roster/save resource data
- Build or tooling: possible small extraction tool or documented external tool invocation
- Documentation: feature doc, fixture notes, and tracker updates

## Validation Plan

Validate by extracting or documenting at least one representative fixture path and confirming the extracted byte sizes/records match the legacy source expectations. If harness support is added, run:

```bash
make -C harness test
```

## Open Questions

- Do we have or need a checked-in legacy roster fixture?
- Should fixture data be committed as binary bytes, generated text, or generated during tests?
- Can modern macOS tooling read the relevant resource fork/container shape reliably enough?
