# F-012: Save Resource Fixture Extractor

## Objective

Create a read-only fixture extraction path for the legacy roster/save resource data needed by future persistence characterization.

## User Value

Save compatibility is hard to reason about from source inspection alone. A fixture extractor gives maintainers stable examples of `PRTY`, `ROST`, current Sosaria `MAPS`, current Sosaria `MONS`, and mutable `MISC` data before a modern save adapter is designed.

## Non-Goals

- Do not implement a new save format.
- Do not write or migrate user save files.
- Do not replace Resource Manager loading in the app.
- Do not require final compatibility policy decisions.

## Acceptance Notes

- Identify the minimum fixture data needed for party, roster, current Sosaria map, current Sosaria monsters, and mutable `MISC` resources.
- Choose a read-only extraction approach for legacy roster resource files, using a tool, parser, or documented manual fixture path.
- Preserve byte-level data in a form suitable for harness tests or future adapter tests.
- Document limitations, especially around resource forks and modern filesystem behavior.

## Result

Status: complete as of 2026-06-05.

Added a read-only classic resource parser in `Core/src/u3_resource.c`, with declarations in `Core/include/u3_resource.h`. The parser accepts an in-memory classic resource-file byte buffer, validates the resource header, type list, reference lists, names, and data ranges, and returns resource records as pointers into the original byte buffer. It does not call Resource Manager APIs and does not write or migrate resources.

Added `harness/resource_fixture_tests.c` to validate the parser against the checked-in `Resources/English.lproj/MainResources.rsrc`. The test confirms this archive is parseable as a data-fork resource file and preserves the default save-template bytes needed to create a new roster file:

| Source record | Length | Role |
| --- | --- | --- |
| `PRTY` 500 | 64 bytes | Default active party template copied to mutable `PRTY` 400 |
| `ROST` 500 | 1280 bytes | Default 20-character roster template copied to mutable `ROST` 400 |
| `MAPS` 420 | 4101 bytes | Default Sosaria map template copied to mutable `MAPS` 419 |
| `MONS` 420 | 256 bytes | Default Sosaria monster template; new-file legacy path creates mutable `MONS` 419 as zero-filled data instead |
| `MISC` 400 | 16 bytes | Moongate-location table template copied to mutable `MISC` 500 |
| `MISC` 401 | 11 bytes | Type-initial table template copied to mutable `MISC` 501 |
| `MISC` 402 | 11 bytes | Weapon-use table template copied to mutable `MISC` 502 |
| `MISC` 403 | 11 bytes | Armour-use table template copied to mutable `MISC` 503 |
| `MISC` 404 | 64 bytes | Location table template copied to mutable `MISC` 504 |
| `MISC` 405 | 16 bytes | Experience table template copied to mutable `MISC` 505 |

The harness also confirms the runtime mutable save records are not present in `MainResources.rsrc`: `PRTY` 400, `ROST` 400, `MAPS` 419, `MONS` 419, and `MISC` 500 through 505. Those records belong to the user roster resource file created by `OpenRstr`.

No checked-in changed-save roster fixture exists. Future persistence characterization that needs modified party, roster, current Sosaria, or mutable table state still needs a sample `Ultima III Roster` resource file or an explicit generator that builds the legacy new-file layout from the default templates.

## Affected Systems

- Legacy source files: reference only, primarily `Sources/UltimaMisc.c` and `Sources/UltimaMain.c`

- Documentation: feature tracker, harness README, validation strategy, modernization inventory, and bug tracker

```bash
make -C harness test
```

Validation passed with the integrated harness.

## Open Questions

- Resolved for the checked-in archive: `MainResources.rsrc` can be parsed directly as a classic data-fork resource file without `DeRez`.
- Open for changed-save compatibility: no checked-in legacy roster fixture is available, so F-012 only characterizes bundled default save templates.
- Open for future adapter design: generated fixtures may be sufficient for new-game defaults, but legacy import/export testing still needs real or synthesized user roster files.
- Follow-up bug: `B-002` tracks the mismatch between 11-byte bundled `MISC` table resources and legacy 12-byte copy loops for type, weapon-use, and armour-use tables.
