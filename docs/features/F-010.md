# F-010: Save and Resource Format Inventory

Status: Complete

Last Updated: 2026-06-04

## Objective

Inventory the legacy save, roster, map, monster, talk, and resource data formats that affect modernization.

## User Value

Persistence and resource loading are major blockers for a modern app shell. Before choosing replacement formats or compatibility guarantees, the project needs a clear map of which legacy resource types are still runtime-critical, which data is mutable, and which formats must remain readable.

## Non-Goals

- Do not implement a new save format.
- Do not convert `.rsrc` data or bundled assets.
- Do not remove Resource Manager code from the legacy app.
- Do not decide final compatibility policy unless the evidence is already strong enough.

## Acceptance Notes

- Inventory save and roster functions such as `GetParty`, `PutParty`, `GetRoster`, `PutRoster`, `GetSosaria`, `PutSosaria`, `PushSosaria`, and `PullSosaria`.
- Identify resource types, approximate record ownership, mutable versus read-only data, and known file locations or path assumptions.
- Document compatibility risks for classic resource forks, `FSSpec`/`FSRef`, bundled `.rsrc` data, and writable game state.
- Recommend one or more follow-up milestones for format extraction, fixtures, or adapter design.

## Decision Checkpoint

No durable format decision was accepted during this inventory. The evidence is strong enough to say that persistence and resource loading need an adapter boundary, but not strong enough to choose a final save format, import policy, or converted asset representation.

Keep the existing policy from `DECISION_LOG.md`: legacy code and assets remain the behavioral reference, and broad format conversion should wait for characterization or explicit adapter work.

## Inventory Summary

Legacy persistence uses a writable Resource Manager file named `Ultima III Roster`, opened or created under the classic Preferences folder by `OpenRstr` in `Sources/UltimaMisc.c`. On first run, the file receives mutable `PRTY`, `ROST`, current Sosaria `MAPS`, current Sosaria `MONS`, `PREF`, and duplicated mutable `MISC` resources. `MAPS`, `PRTY`, `ROST`, and `MISC` are copied from bundled defaults in the new-file path; current Sosaria `MONS` is initially created as a zero-filled 256-byte resource.

The bundled `Resources/English.lproj/MainResources.rsrc` remains runtime-critical. It is referenced by the Xcode `Rez` build phase and supplies classic resources used by gameplay setup, maps, monsters, talk tables, combat screen fixtures, demo data, sign data, legacy sound data, old string lists, dialogs, and default save-state templates.

Modernized bundle assets already exist beside the resource archive: plist-backed localized strings, PNG/GIF/JPEG graphics, WAV/MP3 sound effects, MOV music, PNG cursors, NIB UI, and PDF/image reference material. These are mostly read-only bundle assets, not mutable save state.

## Save Flow Notes

- Startup calls `OpenRstr`, then `DoSplashScreen` loads mutable tables with `GetMiscStuff(100)`, `GetRoster`, and `GetParty`.
- Manual quit/save uses `QuitSave`. It refuses to save unless `Party[3] == 0`, records `xpos`/`ypos` into `Party[4]`/`Party[5]`, then calls `PutRoster`, `PutParty`, and `PutSosaria`.
- `PutParty` also only writes when `Party[3] == 0`. Other flows save at transitions by returning or flushing state to Sosaria first.
- Entering towns, castles, Ambrosia, or other non-Sosaria contexts uses `PushSosaria` before loading another map, then `PullSosaria` or explicit save calls when returning. The in-memory `pushMap` offset math should be characterized before being copied into a new core structure.
- Auto-save paths appear in transitions such as `Routine6E6B`, all-dead handling, and map changes. These paths must be preserved behaviorally before a new persistence adapter changes write timing.
- `LoadUltimaMap` contains an in-place compatibility migration for a current Sosaria map resource with handle size `4100`, resizing it to `4101` and inserting a size byte of `64`.

## File Locations and Path Assumptions

- `OpenRstr` locates the roster file in the classic Preferences folder using `FindFolder(kOnSystemDisk, kPreferencesFolderType, kDontCreateFolder, ...)` and a localized string for `Ultima III Roster`.
- `OpenRstr` resolves aliases with `IsAliasFile`/`ResolveAliasFile`, then opens the file with `FSpOpenResFile(..., fsRdWrPerm)`.
- If the preferences roster file is missing, the code creates a Resource Manager file with creator `Ult3` and type `RSTR`.
- On first-run creation, the code also looks for an old relative `:Roster` file next to the app and copies its fork bytes into the new preferences roster file.
- Bundle resources are split across the classic `.rsrc` archive and copied folders/files. `Ultima3.xcodeproj/project.pbxproj` includes `MainResources.rsrc` in a `PBXRezBuildPhase`, while `Music`, `Sounds`, `Graphics`, `Strings`, `Images`, cursors, the NIB, and `Application.icns` are copied through the resource build phase.

## Compatibility Risks

- Writable resource forks are the central persistence mechanism; a modern app should not depend on Resource Manager write APIs.
- `FSSpec`, `FSRef`, `FindFolder`, `FSpOpenResFile`, `FSpCreateResFile`, `FSReadFork`, and `FSWriteFork` are all migration blockers for Apple silicon-native persistence.
- The save file is not just party and roster data. It also stores current Sosaria map/monster state and mutable `MISC` tables, so an import/export path must treat it as a resource container, not a flat character file.
- Legacy write timing is gameplay-relevant. Manual saves are restricted to outdoors in Sosaria, but auto-save and map-transition paths flush state at specific points.
- Current Sosaria has at least one legacy on-load migration path for 4100-byte resources. Fixtures should include both modernized and pre-migration shapes if old save compatibility is required.
- `MainResources.rsrc` is still required even though many assets have been moved to modern files. Replacing it requires extracting map, monster, talk, combat screen, demo, sign, legacy string/dialog, and default save templates.
- `DeRez Resources/English.lproj/MainResources.rsrc` failed locally with `eofErr (-39)`, so binary resource enumeration remains incomplete. `strings` confirmed resource type/name hints including `MAPS`, `MISC`, `MONS`, `PRTY`, `SGNT`, `TLKS`, `ROST`, `CONS`, `DEMO`, `Sosaria`, `Party`, and `Roster`.

   Build a read-only tool or harness fixture path that can parse or snapshot the roster resource data needed by `PRTY`, `ROST`, `MAPS`, `MONS`, and mutable `MISC` resources.

   Enumerate `MainResources.rsrc` records with a working legacy toolchain or custom parser, then map each resource type/id to its source consumer and converted-data candidate.

3. `F-014: Persistence Adapter Design`
   Decide whether modern saves support exact legacy import/export, best-effort import only, or a new format with no legacy compatibility. Include manual-save and auto-save timing rules.

4. `F-015: Map/Talk/Combat Fixture Characterization`
   Add harness fixtures for non-Sosaria map loads, talk tables, and `CONS` combat screens before replacing Resource Manager loading in gameplay setup.
