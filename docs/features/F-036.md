# F-036: Save Workflow and Autosave Policy

## Objective

Define and implement the first user-facing save workflow beyond smoke commands, including when current native save changes are written automatically versus manually.

## User Value

Once movement and gameplay state mutate the current document, players need a predictable way to keep progress. This feature turns the existing smoke write path into a clearer gameplay save policy before transition and encounter systems add more mutation points.

## Non-Goals

- Do not implement exact classic roster export.
- Do not implement a full document browser or multi-slot save manager unless selected as a separate UI feature.
- Do not save uncharacterized location states or destination maps without an explicit byte contract.
- Do not move persistence ownership into AppKit; the shell provides locations and the adapter owns serialization/write validation.

## Dependencies

- F-030 provides current native save load/write.
- F-031 and F-033 introduce gameplay mutations that should persist.
- F-035 may expose the first non-overworld state that affects save timing decisions.

## Acceptance Notes

- The project documents a first autosave/manual-save policy for current native save documents.
- Smoke-oriented menu labels are replaced or supplemented with user-facing save/load commands when the policy is accepted.
- Manual save validates that the current state is safe to write under the documented restriction.
- If autosave is added, it is limited to characterized safe states and failures preserve the previous valid saved document.
- Manual validation covers save, quit, relaunch, and load after at least one gameplay mutation.

## Affected Systems

- Portable core: likely `Core/include/u3_save.h`, `Core/src/u3_save.c`
- Modern shell: `ModernShell/Sources/AppDelegate.swift`, `ModernShell/Sources/ShellSaveAdapter.swift`, menu command labels and status handling
- Modernization harness: `harness/save_fixture_tests.c`, plus Swift save adapter tests
- Documentation: `ModernShell/README.md`, feature tracker, and possibly `DECISION_LOG.md` if the policy is durable

## Validation Plan

Add tests for valid manual save, unsafe-state rejection, previous-save preservation after failed writes, and reload after gameplay mutations. Add a decision-log entry if the save policy is broader than a single feature contract.

Manual validation:

1. Run `swift run Ultima3ModernShell`.
2. Load or create a current save with an active party.
3. Perform a documented gameplay mutation, such as movement or turn-economy state change.
4. Use the user-facing save command.
5. Quit and relaunch.
6. Load the saved document and confirm the mutation persisted.
7. Exercise any documented unsafe-state case and confirm the shell reports rejection without corrupting the prior save.

## Open Questions

- Resolved: autosave waits until location-transition and non-overworld persistence timing is characterized. The first policy is manual save only.
- Resolved: the user-facing commands are Game > New Game, Game > Save, and Game > Load Game.
- Resolved: the policy is durable and is recorded in `DECISION_LOG.md`.
