# F-058: Combat Presentation and Turn Feedback Pass

## Objective

Improve combat feedback for already-supported command paths with bounded presentation events, without changing combat rules.

## User Value

Combat status currently proves behavior, but richer feedback will make turns easier to follow as more actions, spells, and outcomes are added.

## Non-Goals

- Do not change combat resolution rules.
- Do not introduce a final renderer or animation engine decision.
- Do not add blocking legacy-style pauses that stall the shell event loop.
- Do not implement new combat commands solely for presentation.

## Dependencies

- F-050 and F-051 provide player and monster turn result fields.
- F-052 through F-055 may add victory, flee, ranged, spell, or defeat events that presentation can reuse.
- F-017 defines the portable render-frame contract.

## Acceptance Notes

- Combat result contracts expose enough bounded presentation metadata for hit flashes, projectile traces, victory/flee/defeat status, or short-lived overlay commands selected for this slice.
- The shell maps presentation metadata into AppKit-rendered frames or status/audio without platform calls from Core.
- Existing deterministic combat tests remain stable.
- Presentation effects do not mutate save data or alter combat decisions.

## Affected Systems

- Portable core: optional presentation/result metadata for existing combat contracts
- Modern shell: render/status/audio mapping for transient effects
- Renderer adapter: bounded display of selected combat feedback
- Modernization harness and Swift tests: deterministic event metadata and frame/status output
- Documentation: `ModernShell/README.md`, feature tracker

## Validation Plan

Cover selected presentation metadata, no-rule-change invariants for combat results, deterministic render/status/audio output, and no document mutation. Run `make -C harness test`, `swift test`, and the Core boundary scan.

Manual validation: enter combat, perform supported attacks and monster turns, confirm feedback is visible and does not alter save or combat outcomes.

## Open Questions

- Should this pass wait until victory/flee/spell/ranged flows are complete, or land earlier to improve manual validation?
- Which presentation primitive is most useful first: hit flash, projectile trace, status overlay, or message log?
