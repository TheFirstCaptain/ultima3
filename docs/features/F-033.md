# F-033: Overworld Turn Economy Slice

## Objective

Add the first save-backed overworld turn consequences after movement and passability, focused on deterministic per-step state updates such as food or movement counters.

## User Value

Movement should start feeling like actual gameplay instead of free viewport scrolling. A narrow turn-economy slice gives each accepted movement command a visible state consequence while keeping random encounters, transitions, and combat out of scope.

## Non-Goals

- Do not implement full monster movement or encounter generation.
- Do not implement town, dungeon, castle, Ambrosia, or combat transitions.
- Do not implement every legacy hunger, poison, vehicle, or time-of-day side effect in this slice.
- Do not let the shell mutate save bytes directly; portable core or persistence-domain helpers must own rule-level mutations.

## Dependencies

- F-031 should provide save-backed party position and rendering.
- F-032 should provide explicit passable versus blocked movement results.
- F-030 provides current native save ownership, load, and writeback; this feature must add or reuse explicit mutation helpers for the selected state update.

## Acceptance Notes

- One deterministic turn-economy family is selected and documented before implementation, with food or a movement counter preferred as the first target.
- Accepted movement applies the selected state update to the current native save document.
- Blocked movement behavior is explicitly chosen: either no turn cost or the legacy-characterized cost.
- The portable result reports the state delta in a testable form for shell status and future message/audio adapters.
- The shell can inspect or report the changed value after movement and after save/relaunch.

## Affected Systems

- Legacy source files: `Sources/UltimaMain.c`, `Sources/UltimaMisc.c`
- Portable core: likely `Core/include/u3_overworld.h`, `Core/src/u3_overworld.c`, and `Core/include/u3_save.h`
- Modern shell: tick-driven movement path and status/inspection path
- Modernization harness: `harness/overworld_movement_tests.c`, `harness/save_fixture_tests.c`
- Documentation: feature tracker, `ModernShell/README.md`, and any durable byte-contract notes

## Validation Plan

Add fixture-backed harness tests for accepted movement state deltas, blocked movement cost behavior, save-domain byte mutation, and persistence round trip. Add Swift tests if the shell adapter owns any additional current-document transaction behavior.

Run:

```bash
make -C harness test
swift test
rg -n "AppKit|SwiftUI|Metal|AVFoundation|Foundation|SDL|NS[A-Z][A-Za-z]+\\b|NSURL|Sandbox" Core
```

Manual validation:

1. Run `swift run Ultima3ModernShell`.
2. Load or create a current save with an active party.
3. Inspect the selected turn-economy value.
4. Move onto passable terrain and confirm the value changes as expected.
5. Attempt a blocked movement and confirm the documented blocked-move behavior.
6. Write the save, relaunch, load the saved document, and confirm the changed value persists.

- Should this slice expose the value through `Inspect Party/Roster`, a new debug inspector, or game-host status only?
