# F-002C: Pascal String Helper Characterization

## Objective

Characterize byte-level Pascal string helper behavior from `Sources/UltimaText.c`.

## User Value

Legacy strings appear throughout text, messages, resources, and UI. Exact string behavior should be preserved before conversion to modern string types.

## Non-Goals

- Do not modernize the text rendering system.
- Do not convert localized string resources.
- Do not characterize font measurement or wrapping functions that require rendering APIs unless a shim exists.

## Acceptance Notes

- Characterization covers `StringLocation`, `SearchReplace`, `IsNewline`, and `AddString`, or documents deferrals.
- Tests assert exact Pascal string length bytes and content bytes.
- Quirky loop bounds and off-by-one behavior are preserved unless intentionally documented.

## Result

Status: complete as of 2026-05-30.

The harness lives in `harness/pascal_string_helpers_tests.c` and covers:

- `StringLocation`
- `SearchReplace`
- `IsNewline`
- `AddString`

It uses synthetic Pascal string fixtures and local `BlockMoveData` / `BlockMove` shims backed by `memmove`. It intentionally preserves observed legacy quirks:

- `StringLocation` misses matches at the last possible start position.
- `StringLocation` misses equal-length full-string matches.
- `SearchReplace` uses the same final-position loop bound, so terminal matches may remain unchanged.
- `SearchReplace` can alter later matching behavior after a length-changing replacement because it mutates the source while scanning.
- `AddString` has no bounds check; the harness uses extra backing storage for the length-byte wrap fixture to avoid corrupting the test process.
