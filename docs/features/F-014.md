# F-014: Persistence Adapter Design

## Objective

Decide the first modern persistence adapter direction for party, roster, current Sosaria state, mutable tables, and legacy save compatibility.

## User Value

The app shell and portable core need a clear persistence boundary before save/load behavior moves away from Resource Manager APIs. This design should prevent accidental behavior drift around manual saves, auto-save timing, and legacy roster data.

## Non-Goals

- Do not implement the persistence adapter.
- Do not migrate user saves.
- Do not remove legacy `OpenRstr`, `PutParty`, `PutRoster`, or `PutSosaria` code.
- Do not choose unrelated app-shell, renderer, or audio architecture.

## Acceptance Notes

- Compare compatibility options: exact legacy import/export, best-effort import only, or new-format-only.
- Define the first adapter boundary in terms of data loaded/saved, write timing, error handling, and ownership.
- Account for manual save restrictions, auto-save calls, map-transition `PushSosaria`/`PullSosaria`, and current Sosaria migration behavior.
- Record an accepted or deferred decision in `DECISION_LOG.md`.
- Propose follow-up implementation and characterization milestones.

## Result

Status: complete as of 2026-06-05.

F-014 accepts a first persistence direction: native modern saves should use a canonical modern save document, while legacy `Ultima III Roster` compatibility should start as best-effort import after mutable roster fixtures exist. The accepted decision is recorded in `DECISION_LOG.md` under "Persistence Adapter Uses Modern Native Save With Legacy Import."

### Compatibility Options

| Option | Result | Rationale |
| --- | --- | --- |
| Exact legacy import/export | Deferred | The project can parse classic data-fork resource files and default templates, but it still lacks a real mutable `Ultima III Roster` fixture. Exact export would risk producing files that appear compatible but miss resource-map, resource-fork, or write-timing details. |
| Best-effort legacy import plus native modern saves | Accepted | This preserves a practical path for historical saves while avoiding Resource Manager write APIs in the modern app. It also lets adapter tests grow from F-012/F-013/F-015 fixtures before promising exact round-trip compatibility. |
| New-format-only | Rejected for the first adapter | It would simplify implementation but discard feasible compatibility with historical roster/save data and reduce confidence that modern persistence preserves legacy behavior. |

### Adapter Boundary

The first adapter should own a modern save document with explicit records matching legacy save domains:

| Domain | Legacy source | Adapter ownership |
| --- | --- | --- |
| Active party | `PRTY` 400, copied to/from `Party[1..64]` | Store exactly 64 party bytes plus adapter metadata needed to interpret current location and save eligibility. |
| Character roster | `ROST` 400, 20 records of 64 bytes | Store 1280 roster bytes as a byte-preserving block until a stronger character model is characterized. |
| Current Sosaria map | `MAPS` 419 | Store map size, map bytes, and whirlpool bytes as an explicit current-Sosaria record. Preserve the legacy 4100-byte-to-4101-byte migration as an import concern. |
| Current Sosaria monsters | `MONS` 419 | Store exactly 256 monster bytes. New-game defaults should follow the legacy first-run behavior where current Sosaria monsters start as zero-filled mutable data. |
| Mutable gameplay tables | `MISC` 500..505 | Store each table as a separate byte record rather than merging immediately. F-027 resolves the length policy for native documents as `16, 11, 11, 11, 64, 16`, while future exact legacy import can still add explicit compatibility if real roster evidence requires it. |
| Legacy preferences | `PREF` 400 | Preserve as optional compatibility metadata if imported, but do not make it part of app preference state until a consumer is found. |

The adapter should be independent of app-shell storage APIs. The modern shell should pass a save-file URL/path into the adapter; the adapter should not call AppKit, Carbon, Preferences-folder, sandbox, or application-support APIs directly.

### Write Timing and Behavior

- Startup should load or initialize persistence before gameplay state setup that currently calls `GetMiscStuff(100)`, `GetRoster`, and `GetParty`.
- Manual save should preserve the legacy `QuitSave` rule: durable writes are allowed only while `Party[3] == 0`; before writing, party position bytes must be synchronized from `xpos` and `ypos` into `Party[4]` and `Party[5]`.
- `PutParty` semantics should remain outdoor-only for durable party writes, matching the legacy `Party[3] == 0` guard.
- Transition saves that currently call `PutRoster`, `PutParty`, and `PutSosaria` before `PushSosaria` or map loading should remain explicit adapter flush points.
- `PushSosaria`/`PullSosaria` should remain in-session snapshot behavior at the first boundary. They should not become durable writes unless the legacy path explicitly calls save functions.
- Auto-save paths such as dungeon return, all-dead handling, Ambrosia/whirlpool transitions, and reset/resurrect flows should be modeled as explicit adapter operations rather than hidden state mutation.
- Writes should be atomic at the adapter boundary: validate and serialize a complete document, then replace the previous durable document only after the new document is complete.

### Error Handling

- Load failures should fail closed and report an adapter error rather than partially populating game state.
- Import should preserve the original legacy bytes in memory until conversion succeeds.
- Native save writes should retain the last valid save document if serialization or replacement fails.
- Unknown legacy records should be retained as compatibility metadata during import where practical, but the first adapter should only interpret the explicit save domains above.

## Open Questions

- Resolved for first adapter: legacy save compatibility starts as best-effort import, not exact import/export and not new-format-only.
- Resolved for first adapter: mutable gameplay tables remain separate byte records inside the modern save document.
- Deferred to app-shell work: concrete save-file location for non-App Store and sandboxed builds. The adapter accepts a caller-supplied storage location.
- Still open: real mutable `Ultima III Roster` fixture coverage for import tests.
- Still open: exact legacy export, pending mutable roster fixtures and resource-container round-trip validation.
- Still open: whether `PREF` 400 has any runtime consumer or should remain compatibility metadata only.
