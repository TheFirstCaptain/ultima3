# F-014: Persistence Adapter Design

## Objective

Decide the first modern persistence adapter direction for party, roster, current Sosaria state, mutable tables, and legacy save compatibility.

## User Value

The app shell and portable core need a clear persistence boundary before save/load behavior moves away from Resource Manager APIs. This design should prevent accidental behavior drift around manual saves, auto-save timing, and legacy roster data.

## Non-Goals

- Do not implement the persistence adapter.
- Do not migrate user saves.
- Do not remove legacy `OpenRstr`, `PutParty`, `PutRoster`, or `PutSosaria` code.
- Do not choose unrelated app-shell, renderer, or audio architecture.

## Acceptance Notes

- Compare compatibility options: exact legacy import/export, best-effort import only, or new-format-only.
- Define the first adapter boundary in terms of data loaded/saved, write timing, error handling, and ownership.
- Account for manual save restrictions, auto-save calls, map-transition `PushSosaria`/`PullSosaria`, and current Sosaria migration behavior.
- Record an accepted or deferred decision in `DECISION_LOG.md`.
- Propose follow-up implementation and characterization milestones.

## Affected Systems

- Legacy source files: reference only, primarily `Sources/UltimaMisc.c`, `Sources/UltimaMain.c`, and app startup/quit paths
- Modernization harness: likely future adapter tests, no immediate changes required
- Resources/assets: save/resource fixtures from F-012 if available
- Build or tooling: none expected
- Documentation: decision log, feature doc, modernization strategy, and tracker updates

## Validation Plan

This is a design milestone. Validate against F-010 inventory evidence, any F-012 fixture evidence available, and local source references for save timing. If external macOS filesystem or app-container facts are needed, verify against official documentation before recording a durable decision.

## Open Questions

- Is legacy save compatibility required, best-effort, or intentionally out of scope?
- Where should modern saves live for non-App Store and possible sandboxed builds?
- Should mutable gameplay tables remain separate records or be wrapped in a single modern save document?
