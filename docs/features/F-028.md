# F-028: Persistent Character Creation

## Objective

Turn the F-026 character creation spike into a save-domain mutation path that writes a valid created character into a native `ROST` slot.

## User Value

Players need created characters to persist beyond a status message before the modern shell can support party assembly, new game setup, or real save/load workflows.

## Non-Goals

- Do not make SwiftUI own shell lifecycle, the primary game surface, or first-hop input.
- Do not write classic Resource Manager roster files.
- Do not add active party selection in this feature.
- Do not implement every legacy character creation affordance, such as random names or help artwork, unless needed by the mutation contract.

## Dependencies

- F-024 exposes party and roster summaries.
- F-026 validates character candidates through portable core rules.
- F-027 should settle mutable `MISC` length policy before broader save-domain writes expand.
- F-030 should provide the shell-held current native save document before this feature wires mutation into the app shell. Portable roster encoding can be characterized independently first.
- F-014 and F-020 define the native save document direction.

## Acceptance Notes

- A valid character candidate can be encoded into an empty roster slot in a native save-domain document.
- The encoded record preserves legacy first-character defaults: status good, current/max HP 100, food 150, gold 150, cloth armour, dagger weapon, and readied equipment bytes.
- The shell-created candidate updates inspected party/roster state after acceptance.
- Cancelled or invalid candidates do not mutate the save-domain document.
- The mutation API uses portable C data and does not expose AppKit, SwiftUI, Foundation, or filesystem types through `Core/`.

## Implementation Notes

- Added `u3_character_add_to_roster`, which validates a candidate, finds the first empty roster slot using the same `record[0] <= 22` availability rule as the roster loader, and writes the fixed 64-byte legacy roster record.
- The first empty-slot policy is accepted for this slice; slot selection and editing occupied slots remain future roster-management work.
- The written record preserves the legacy `CreateCharacter` defaults: status `G`, HP/max HP `100`, food `150`, gold `150`, cloth/readied armour bytes, and dagger/readied weapon bytes.
- `ShellCharacterCreationAdapter.persist` requires a loaded, save-domain-valid native document before patching the fixed-length `ROST` record.
- `CharacterCreationView` now passes the accepted draft to the shell; `ShellSmokeState` requires a loaded current save before persisting and refreshes party/roster status from the updated document.
- Active party membership is intentionally unchanged; F-029 remains responsible for selecting party members from occupied roster entries.

## Validation Plan

Add harness coverage for roster-slot encoding, occupied-slot rejection, invalid candidate rejection, and legacy default bytes. Add Swift tests for shell acceptance, cancellation, and inspected roster summary after creation.

Run:

```bash
make -C harness test
swift test
rg -n "AppKit|SwiftUI|Metal|AVFoundation|Foundation|SDL|NS[A-Z][A-Za-z]+\\b|NSURL|Sandbox" Core
```

Manual validation:

1. Run `swift run Ultima3ModernShell`.
2. Choose `Game > Create Character...`.
3. Enter a valid candidate and accept.
4. Choose `Game > Inspect Party/Roster`.
5. Confirm the created character appears in the roster status.
6. Repeat with Cancel and confirm no new roster mutation occurs.
