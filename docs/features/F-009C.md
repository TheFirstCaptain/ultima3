# F-009C: Monster Combat Action Resolution Core

## Objective

Characterize bounded monster-turn side effects from `Sources/UltimaSpellCombat.c`, including movement, shooting, spell targeting, poison, pilfering, hit/miss, and player damage.

## User Value

Monster turns define combat difficulty and status effects. Extracting this slice after player damage and attack handling will make combat behavior testable without Carbon input, QuickDraw rendering, or QuickTime sound.

## Non-Goals

- Do not port the full `Combat()` loop.
- Do not port combat setup, `GetScreen`, victory, player command input, or spell selection.
- Do not implement platform rendering, audio, text, or preference reads in the core.
- Do not change poison, pilfer, monster movement, or damage behavior unless documented.

## Acceptance Notes

- Characterize `Poison`, `Pilfer`, `FigureNewMonPosition`, shooting, monster spell targeting, hit chance, and player damage as explicit state transitions.
- Preserve relevant `gMonType`, `g835E`, armour, exotic armour, party size, and character-status behavior.
- Model text, sound, projectile path, hit animation, death checks, and redraw as events.
- Add deterministic harness coverage with supplied random values.

## Affected Systems

- Legacy source files: reference only, primarily `Sources/UltimaSpellCombat.c`
- Modernization harness: likely new monster action tests
- Resources/assets: none
- Build or tooling: likely `Core/include/u3_combat.h`, `Core/src/u3_combat.c`, and `harness/Makefile`
- Documentation: feature tracker, harness README, validation strategy

## Validation Plan

Run:

```bash
make -C harness test
```

## Open Questions

- Should monster movement and attack resolution be split into separate features if the event model becomes large?
- Should player HP/status mutation share a future party-state core module rather than live only in combat?
