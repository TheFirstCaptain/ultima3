# F-046: Dungeon Special Tile Effects Slice

## Objective

Implement the first non-interactive dungeon special tile effects through portable result contracts: strange wind extinguishing light, traps, gremlin food theft, and writing lookup/display.

## User Value

Dungeon tiles should produce recognizable consequences instead of behaving like ordinary floor. This adds hazard and message feedback while staying short of larger interactive flows.

## Non-Goals

- Do not implement fountains, marks, Time Lord encounters, or chest opening.
- Do not implement full combat encounters or trap-disarm character prompting beyond the minimum characterized first slice.
- Do not persist in-dungeon state to disk.
- Do not add new artwork or replace the existing perspective renderer.

## Dependencies

- F-043 provides active dungeon command routing and current-cell evaluation.
- F-044 provides light state so wind can extinguish it.
- F-045 defines shared post-command mechanics and clarifies ordering for tile effects versus random encounters.
- F-024 exposes party/roster state needed for food and HP/status mutation.

## Acceptance Notes

- Portable tile-effect logic accepts explicit dungeon tile value, party/roster state inputs, deterministic random values where needed, and returns bounded mutation/event data.
- Strange wind clears the dungeon light counter and redraws/reports explicit feedback.
- Trap handling clears the trap tile when legacy behavior does, reports disarm success/failure, and applies bounded damage events without platform audio or UI calls from Core.
- Gremlin handling selects an eligible living party member deterministically from supplied random input and decrements food without underflow.
- Writing resolves the characterized dungeon talk line for the current level and reports a printable bounded message.
- Party and roster mutations such as food, HP, and status changes apply to the current in-memory native document, mark it as unsaved, and can become durable only after the dungeon exits to Sosaria and the existing manual Save command succeeds.
- Dungeon-session mutations such as clearing a trap tile apply only to the adapter-owned transient dungeon map copy and are discarded when the dungeon session is replaced unless a later feature explicitly adds durable dungeon-map persistence.
- Unsupported special tiles return explicit deferred/unsupported results without mutating session or save state.

## Affected Systems

- Legacy source files: `Sources/UltimaDngn.c` `dngnotcombat` cases 3, 4, 6, and 8
- Portable core: dungeon tile-effect contracts and party/roster mutation helpers as needed
- Modern shell: status/audio/redraw mapping and session mutation application
- Resources/assets: existing dungeon `TLKS` records for writing
- Modernization harness and Swift tests: special tile fixtures, mutation ordering, malformed data rejection
- Documentation: `ModernShell/README.md`

## Validation Plan

Cover wind, trap success/failure, food decrement and zero clamp, current-document dirty-state marking for party/roster mutations, transient-only dungeon tile clearing, writing lookup, unsupported tile values, malformed talk records, redraw/status/audio fields, save rejection while effects occur in dungeon mode, and return-to-Sosaria save/reload behavior proving party/roster mutations persist while transient dungeon tile changes do not. Run `make -C harness test`, `swift test`, and the Core boundary scan.

Manual validation: navigate or seed a known dungeon special tile, trigger each supported effect, confirm visible status/audio/redraw behavior, confirm party/roster changes remain pending in the current document while in dungeon mode, confirm transient tile changes are not treated as durable save data, return to Sosaria, save, reload, and confirm only party/roster mutations persisted.

## Open Questions

- Resolved for this slice: trap disarm uses the first living active party member from the current native `PRTY`/`ROST` records. Prompt-backed character selection remains deferred to F-047.
- Resolved for this slice: writing displays through the existing status overlay using bounded decoded `TLKS` text. A dedicated longer-text panel remains deferred.
