# F-051: First Monster Combat Turn Slice

## Objective

Run a bounded monster action phase after handled player combat commands using portable monster-action results.

## User Value

Combat begins to feel reciprocal. Monsters can move, attack, shoot, poison, pilfer, or report deferred actions through explicit state results instead of remaining static.

## Non-Goals

- Do not implement the full spell system or every monster special case if a smaller first set is needed.
- Do not implement victory, loot, or combat exit.
- Do not add animation timing, projectile visuals, or blocking pauses beyond status/audio events.
- Do not make combat state durable.

## Dependencies

- F-048 and F-049 provide active combat sessions and renderable occupant state.
- F-050 provides player command handling and post-command sequencing.
- F-009C extracted bounded monster combat action resolution.
- F-024 and F-030 expose party/roster data in the current native document.

## Acceptance Notes

- A handled player combat command can request a deterministic monster action phase.
- Monster actions use explicit random inputs in tests and runtime random values in the shell.
- Movement, attack, poison, pilfer, shooting, and unsupported/deferred actions return bounded result/event data without platform calls from Core.
- HP/status/inventory mutations apply to the current in-memory native document, mark it unsaved, and remain durable only after later return to Sosaria and manual Save.
- Combat session redraw/status/audio update after monster actions.
- Dead or invalid party/monster state suppresses unsafe actions and reports explicit status.

## Affected Systems

- Legacy source files: `Sources/UltimaSpellCombat.c` monster turn helpers including `Poison`, `Pilfer`, `Shoot`, movement/action selection
- Portable core: monster action sequencing around existing F-009C contracts
- Modern shell: post-player-command monster phase
- Save/resource adapters: roster/inventory/status mutation application
- Modernization harness and Swift tests: deterministic monster action cases and mutation durability
- Documentation: `ModernShell/README.md`

## Validation Plan

Cover deterministic monster movement, melee hit/miss/damage, poison, pilfer, shooting/deferred spell paths, dead/invalid target suppression, redraw/status/audio fields, document dirty-state marking, and save rejection while combat remains active. Run `make -C harness test`, `swift test`, and the Core boundary scan.

Manual validation: enter combat, issue player commands until monster actions occur, confirm status/audio/redraw behavior, inspect party/roster changes, and confirm Save remains rejected.

## Open Questions

- Should monster spell casting be deferred entirely, or represented as explicit unsupported spell metadata in the first monster-turn slice?
- Should monster turns process one monster per tick or a complete legacy-style monster phase per handled player command?
