# F-048: Combat Session Entry From Encounter Requests

## Objective

Turn existing deferred encounter metadata into a validated transient combat session without starting the full combat loop.

## User Value

Dungeon encounters should stop being status-only. This slice proves the shell can cross from exploration into a combat-specific session while keeping combat setup, resources, and return context bounded.

## Non-Goals

- Do not implement player combat commands, monster turns, spells, victory, loot, or combat exit.
- Do not make combat state durable in the native save document.
- Do not replace the renderer or add new combat artwork.
- Do not support every overworld, town, or dungeon combat source if a first Dungeon Doom encounter is enough for validation.

## Dependencies

- F-045 provides deterministic dungeon encounter request metadata.
- F-047 may add more dungeon interactions first, but is not required for combat entry.
- F-015 characterized representative `CONS` combat-screen fixtures.
- F-022 and F-037 provide resource-backed frame/session patterns.

## Acceptance Notes

- A dungeon encounter request can initialize a transient combat session from explicit metadata: source mode, monster type, marker tile, combat screen resource id, party context, and return session.
- The shell validates and copies the selected `CONS` combat-screen record through adapter-owned resource loading before changing active mode.
- Combat entry preserves the exact dungeon session needed to return later, including map copy, level, position, heading, light, and retained Sosaria context.
- Save remains rejected while combat is active.
- Unsupported or malformed combat-screen data reports explicit status without corrupting the dungeon session or current save document.
- Initial combat session state includes enough explicit data for later rendering and turn slices: combat screen id, party slots, monster table/type metadata, and source session identity.

## Affected Systems

- Legacy source files: `Sources/UltimaDngn.c` encounter handoff; `Sources/UltimaSpellCombat.c` `Combat` setup; `Sources/UltimaMisc.c` `GetScreen`
- Portable core: combat session descriptor/setup contract as needed
- Modern shell: active mode/session routing and save rejection
- Resource adapter: `CONS` loading/validation for combat sessions
- Modernization harness and Swift tests: combat session setup, malformed resource rejection, dungeon-session preservation
- Documentation: `ModernShell/README.md`, feature tracker

## Validation Plan

Add harness or Swift adapter coverage for valid `CONS` combat-screen loading, invalid/missing `CONS` rejection, session transition without save mutation, save rejection in combat mode, and preservation of the source dungeon session. Run `make -C harness test`, `swift test`, and the Core boundary scan.

Manual validation: trigger or seed a dungeon encounter, confirm combat mode/session status appears, confirm Save is rejected, and confirm failed setup leaves the dungeon session intact.

## Open Questions

- Should the first combat session be entered only from dungeon encounters, or should overworld location encounters share the initial contract immediately?
- Which subset of `CONS` fields should be copied into the first session versus left as raw adapter-owned bytes until F-049?
