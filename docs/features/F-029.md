# F-029: Party Assembly From Roster

## Objective

Add a modern setup flow for selecting active party members from occupied roster entries and writing the result into the native `PRTY` save-domain record.

## User Value

Created characters need to become an active party before the modern shell can move from smoke gameplay toward a playable new-game loop.

## Non-Goals

- Do not implement full legacy organize-menu behavior.
- Do not implement drag-and-drop party reordering unless the first AppKit/SwiftUI panel needs it for usability.
- Do not enter towns, dungeons, combat, or other maps.
- Do not write classic Resource Manager roster files.

## Dependencies

- F-024 exposes decoded party and roster state.
- F-028 should make character creation persist into `ROST`.
- F-030 should provide the shell-held current native save document before this feature wires party assembly into the app shell. Portable `PRTY` mutation can be characterized independently first.
- F-014 remains the persistence decision source of record.

## Acceptance Notes

- The shell can present occupied roster entries in a non-game setup panel.
- The user can select one to four occupied roster slots as the active party.
- The portable mutation path validates party size, occupied slots, duplicate selections, and roster id bounds.
- The native `PRTY` record is updated with active roster ids and initial new-game location/shape bytes.
- The exact `PRTY` byte contract is documented in this feature before implementation is accepted, including party size, roster id slots, shape, map, position, and any initialized sentinel bytes.
- Inspect Party/Roster reports the newly assembled active party.

## Implementation Notes

- Added `u3_party_form_from_roster` as the portable mutation path for forming a party from occupied `ROST` slots.
- The initialized `PRTY` byte contract for this slice is:
  - byte `0`: party shape `0x7E`, matching the legacy new-party ranger/party shape initialization.
  - byte `1`: active party size, `1...4`.
  - byte `2`: location/map state `0`, meaning Sosaria/outdoors.
  - bytes `3...4`: new-game Sosaria position `42,20`.
  - byte `5`: initialized sentinel `255`, matching legacy `Party[6] = 255`.
  - bytes `6...9`: selected roster ids in user-selected order, with unused slots zero-filled.
  - bytes `10...63`: zero-filled for this first setup slice.
- The mutation validates current save-domain party/roster shape, selection count, roster id bounds, duplicate ids, and occupied roster slots before changing output bytes.
- Added `PartyAssemblyView` as a SwiftUI setup panel hosted by the AppKit shell. Selection order is the order in which roster slots are toggled on.
- Replacing an existing active party is supported, but the shell presents an AppKit confirmation before applying the replacement.
- `Game > Assemble Party...` opens the panel; accepted selections update the shell-held current native save document, and `Game > Inspect Party/Roster` reports the new active party.
- On launch, the shell adopts an existing valid saved native document when one is available, so an assembled party persists across app restarts without manually reloading the saved smoke document first.
- The first slice mutates only `PRTY`; it does not update legacy roster membership flags or implement the full organize/disperse behavior.

## Validation Plan

Add harness coverage for party mutation success, empty selection rejection, duplicate rejection, unoccupied roster rejection, and party-size bounds. Add Swift tests for shell panel acceptance/cancel behavior if the implementation is factored for testing.

Run:

```bash
make -C harness test
swift test
rg -n "AppKit|SwiftUI|Metal|AVFoundation|Foundation|SDL|NS[A-Z][A-Za-z]+\\b|NSURL|Sandbox" Core
```

Manual validation:

1. Run `swift run Ultima3ModernShell`.
2. Choose `Game > Load New Game Smoke`.
3. Choose `Game > Assemble Party...`.
4. Select two occupied roster entries in a known order, such as slot 2 then slot 1.
5. Click `Assemble`; if replacing the default active party, confirm replacement.
6. Choose `Game > Inspect Party/Roster`.
7. Confirm active party size and roster ids match the selection, for example `size 2 active 2/1/0/0`.
8. Choose `Game > Write Save Smoke`, quit, relaunch, choose `Game > Inspect Party/Roster`, and confirm the assembled party persisted.
