# F-013: Resource Data Extraction Inventory

## Objective

Enumerate the records in `Resources/English.lproj/MainResources.rsrc` and map each runtime-relevant resource type/id to its legacy consumer and likely converted-data path.

## User Value

The modern port cannot remove `MainResources.rsrc` until the project knows exactly which records are still required. This feature turns the opaque resource archive into an actionable conversion and adapter backlog.

## Non-Goals

- Do not convert all resources to a new format.
- Do not remove the `PBXRezBuildPhase`.
- Do not change bundled assets or source behavior.
- Do not decide final renderer, audio, or persistence formats.

## Acceptance Notes

- Enumerate resource records for known runtime families such as `MAPS`, `MONS`, `TLKS`, `MISC`, `PRTY`, `ROST`, `CONS`, `DEMO`, `SGNT`, `PICT`, `snd `, and legacy string/dialog resources.
- For each runtime-relevant family, identify source consumers and whether the data appears read-only, template data, or save-related.
- Document the toolchain or parser used, including failure modes if legacy tools cannot read the archive.
- Recommend follow-up extraction or characterization targets based on resource risk and modernization value.

## Affected Systems

- Legacy source files: reference only across resource consumers in `Sources/`
- Modernization harness: none required unless fixtures are added
- Resources/assets: read-only inspection of `Resources/English.lproj/MainResources.rsrc`
- Build or tooling: likely `DeRez`, a legacy toolchain, or custom parser investigation
- Documentation: feature doc, modernization inventory, and tracker updates

## Validation Plan

Validate by recording exact commands/tools, inspected resource families, and unresolved enumeration gaps. If a parser or fixture tool is added, validate it against expected resource type names and record counts where available.

## Open Questions

- Why does local `DeRez` fail with `eofErr (-39)` on the checked-in archive?
- Is the archive a data-fork resource file, resource-fork file, or build input that needs different handling?
- Should extracted resource inventories be generated artifacts or checked-in documentation?
