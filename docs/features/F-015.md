# F-015: Map/Talk/Combat Fixture Characterization

## Objective

Add focused characterization fixtures for non-Sosaria map loads, talk tables, and `CONS` combat screen data before replacing Resource Manager-backed loading.

## User Value

Map, talk, and combat setup data are runtime-critical and currently opaque behind Resource Manager calls. Characterization fixtures let future extraction preserve behavior while moving data loading toward portable inputs.

## Non-Goals

- Do not replace all map loading.
- Do not convert every resource record.
- Do not extract full combat orchestration.
- Do not change bundled map, talk, or combat resources.

## Acceptance Notes

- Identify representative non-Sosaria `MAPS`/`MONS`/`TLKS` resources and `CONS` combat screen resources for fixture coverage.
- Capture expected byte layouts and state transitions for `LoadUltimaMap` and `GetScreen` at the selected fixture boundary.
- Add or document harness validation that checks loaded map, monster, talk, tile, character, and combat monster state without platform rendering.
- Record uncovered resource IDs and behavioral assumptions.

## Result

Status: complete as of 2026-06-05.

F-015 extends `harness/resource_fixture_tests.c` with fixture-level characterization for representative `MainResources.rsrc` map, talk, monster, dungeon, and combat-screen records. These tests use the read-only parser from F-012/F-013 and harness-local loaders that mirror the byte-copy boundaries in legacy `LoadUltimaMap` and `GetScreen`; no Resource Manager, rendering, audio, or platform UI calls are involved.

Selected fixtures:

| Fixture | Resources | Why selected | Harness coverage |
| --- | --- | --- | --- |
| Lord British Castle | `MAPS` 400, `MONS` 400, `TLKS` 400 | Representative 64x64 non-Sosaria castle/town map with monster and talk tables | Confirms map size byte `64`, copies 4096 map bytes from offset 1, copies 256 monster bytes, copies 256 talk bytes, and checks representative loaded bytes. |
| Ambrosia | `MAPS` 421, `MONS` 421, `TLKS` 421 | Special non-Sosaria map that legacy `LoadUltimaMap` explicitly includes in the talk-table branch despite `resid > 419` | Confirms the same 64x64 map, monster, and talk layout as town/castle resources. |
| Dungeon Doom | `MAPS` 412, `TLKS` 412 | Representative dungeon map from the `resid > 411 && resid < 419` branch | Confirms a 2048-byte dungeon copy, confirms the legacy `resid < 419` talk-table load, and confirms no bundled `MONS` 412 record is loaded at this fixture boundary. |
| Combat screen A | `CONS` 400 | Representative 200-byte combat screen that satisfies full `GetScreen` offsets | Confirms 121 tile bytes, 8 monster X/Y/tile/HP entries from offsets `0x80`, `0x88`, `0x90`, `0x98`, and 4 character X/Y/tile/shape entries from offsets `0xA0`, `0xA4`, `0xA8`, `0xAC`. |
| Shrine combat/special screen | `CONS` 410 | Representative 121-byte special `CONS` record | Confirms it contains tile bytes only for this boundary and is too short for the full legacy `GetScreen` monster/character offset layout. |

The `CONS` finding is important: F-013 identified 13 `CONS` records, but only records 400 through 408 are 200 bytes and large enough for full `GetScreen` state loading. Records 410 through 413 are 121 bytes, matching the tile grid size but not the monster/character metadata offsets read by `GetScreen`.
