# F-026: Character Creation Flow Spike

## Objective

Draft and spike the first modern character creation flow using shell-owned presentation and portable party/roster state.

## User Value

Character creation is a player-facing setup flow that must eventually replace legacy modal UI while preserving game data constraints.

## Non-Goals

- Do not recreate every legacy dialog exactly.
- Do not make SwiftUI own the primary game surface or deterministic game loop.
- Do not commit to final visual styling for all setup flows.
- Do not write legacy resource files directly.

## Dependencies

- F-024 should expose party and roster state cleanly.
- F-020/F-021 should settle native persistence entry points.
- F-014 remains the save-format decision source of record.

## Acceptance Notes

- The flow can create or edit a character candidate through shell-owned UI.
- Generated state is validated through portable roster/party rules before persistence.
- SwiftUI may be used for the setup panel while AppKit remains shell owner.
- Manual HIL verification confirms the flow can be opened, edited, accepted, and cancelled without affecting the primary shell boundary.

## Implementation Notes

- Added `u3_character_validate_candidate` as a portable first-slice candidate validator.
- First spike is non-persistent: accepting a candidate updates shell status only and does not mutate `ROST`, `PRTY`, or the native save document.
- Validation covers the legacy-dialog constraints visible in `CharacterCreateDialog`: non-empty name, max 12 visible bytes, printable ASCII name bytes, race/class/sex code membership, each attribute in `5..25`, and exactly 50 allocated attribute points.
- Added `Game > Create Character...` as an AppKit-owned menu command that presents a SwiftUI setup panel hosted in an `NSWindowController`.
- The panel edits name, race, class, sex, strength, dexterity, intelligence, and wisdom, validates through Core, accepts valid candidates, and cancels without changing save-domain state.
- Harness coverage validates portable candidate success and rejection paths.
- Swift shell coverage validates adapter summaries/messages and shell accept/cancel status behavior.

## Open Questions

- Resolved for this spike: begin as a non-mutating prototype; roster/native-save mutation remains a later feature after byte-level write rules are characterized.
- Resolved for this spike: characterize only first-dialog candidate rules: name shape, race/class/sex codes, attribute range, and 50-point total.
- Deferred: race/class compatibility, random name generation, initial equipment, HP/MP/food/gold defaults, active party insertion, and durable roster writes.
