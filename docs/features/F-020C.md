# F-020C: Native Save Fixture Builder

## Objective

Define and generate a synthetic new-game native save fixture from characterized default save templates.

## User Value

Before the modern app writes real gameplay saves, maintainers need a canonical native document shape that preserves the legacy save domains accepted by F-014.

## Non-Goals

- Do not implement full gameplay save/load.
- Do not import arbitrary legacy roster files.
- Do not promise exact legacy export.
- Do not collapse byte-preserving save domains into higher-level models prematurely.

## Acceptance Notes

- Native fixture includes explicit records for active party, roster, current Sosaria map, current Sosaria monsters, mutable `MISC` tables, and compatibility metadata.
- Fixture uses F-012 default templates and follows the legacy new-file behavior for zero-filled mutable `MONS` 419.
- Fixture generation is deterministic and covered by adapter tests.
- The format remains adapter-owned and independent of AppKit.

## Affected Systems

- Legacy source files: reference only, primarily `OpenRstr`, `ResetSosaria`, and template-copy paths.
- Modernization harness: native save fixture tests.
- Resources/assets: read-only default templates from `MainResources.rsrc`.
- Build or tooling: possible fixture generator or adapter test helper.
- Documentation: feature doc and tracker.

## Validation Plan

Run fixture-builder tests that assert record presence, byte lengths, default-template copies, and zero-filled current Sosaria monsters. Run full harness and Core boundary scan.

## Open Questions

- Should the first native fixture be binary, JSON plus encoded byte blobs, or a simple chunked document?
