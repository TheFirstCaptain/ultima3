# F-002A: Harness Tooling Decision

## Objective

Choose the minimal tooling and structure for the first legacy behavior characterization harness.

## User Value

A clear tooling decision prevents ad hoc test scaffolding and gives future agents a repeatable way to validate porting work.

## Non-Goals

- Do not implement the full harness in this feature unless explicitly approved after the decision.
- Do not choose the final modern app framework.
- Do not require a legacy SDK or a runnable classic app.

## Acceptance Notes

- Recommended harness approach is documented with rationale.
- Required shims or constraints are identified.
- Validation command is proposed.
- `DECISION_LOG.md` is updated if a tooling decision is accepted.

## Decision

Use a plain C command-line harness under `harness/`, run with:

```bash
make -C harness test
```

Rationale:

- It is independent of Xcode, XCTest, Carbon, QuickTime, legacy SDK paths, NIBs, resource files, and app startup.
- It is small enough for early behavior characterization and future agent handoffs.
- It keeps harness work outside `Sources/`, preserving the legacy source as the reference implementation.
- It can later be replaced or extended with shared portable game-core code once extraction boundaries exist.

Initial constraint: directly compiling all of `Sources/UltimaMisc.c` would require broad shims for unrelated platform dependencies. The first harness may therefore copy narrow deterministic functions into the harness with explicit legacy source references, while tests assert the observed behavior.

Status: complete as of 2026-05-30. The accepted decision is recorded in `DECISION_LOG.md`.

## Affected Systems

- Modernization harness: proposed location and structure
