# F-020D: Save Path and Atomic Write Smoke

## Objective

Have the shell provide a save-document path and have an adapter atomically write and read a small validated smoke document.

## User Value

This proves the modern save path and atomic write behavior before the adapter writes real party, roster, map, or mutable-table data.

## Non-Goals

- Do not save real gameplay state.
- Do not implement auto-save or transition-save timing.
- Do not import legacy roster files.
- Do not choose final user-facing save-management UI.

## Acceptance Notes

- Shell owns the concrete save path.
- Adapter owns serialization and atomic replacement behavior.
- Adapter can round-trip a small smoke document and reject malformed input.
- Failed writes retain the previous valid document.

## Affected Systems

- Legacy source files: reference only, primarily save timing paths in `Sources/UltimaMain.c` and `Sources/UltimaMisc.c`.
- Modern shell: save path provider and status.
- Modernization harness: adapter write/read tests using temporary paths.
- Resources/assets: none.
- Documentation: feature doc and tracker.

## Validation Plan

Run adapter tests for successful write/read, malformed input rejection, and previous-document retention on failed replacement. Build shell, run Core boundary scan, and perform HIL validation of save-path smoke status if exposed.
